# Web UI 初步规划

本文档记录控制台 Web UI 的初步信息架构与演进原则。它不是最终视觉稿；当前先建立可扩展的应用外壳和交互基线，待 M2 告警工作流经过真实使用后，再确定运维总览的最终布局与视觉细节。

## 1. 产品界面目标

登录后的首页不是宣传落地页，也不只是机器列表，而是整个系统的运维总览。用户打开控制台后，应能快速回答四个问题：

1. 当前整体是否健康。
2. 哪些异常最需要关注。
3. 最近发生了什么变化。
4. 当前有哪些安全且可执行的下一步。

界面可以具有鲜明、精致的控制台视觉，但状态识别、信息密度、响应速度和操作反馈优先于装饰效果。视觉上的高级感应主要来自清晰的信息层级、克制的状态色、趋势与时间线、流畅的状态变化，以及一致的组件细节。

## 2. 首页：运维总览

首页最终定位为总体仪表盘，机器列表只作为其中一个区域，并同时保留独立的完整 Fleet 页面。

规划中的首页信息包括：

- Fleet 健康概况：在线、离线、异常、待确认数量。
- 需要关注：按严重程度和时间排序的事件与资源风险。
- 资源概况：CPU、内存、磁盘和可用性的紧凑趋势。
- 机器与服务状态：重点机器、异常服务和健康分布。
- 最近活动：告警、恢复、部署、重启、审批和版本变化。
- Agent 建议：基于当前事实和证据给出的下一步，但不越过审批执行写操作。

M1 阶段以机器状态为主体；M2 加入事件与恢复信息；M3 加入服务、仓库、有限证据和 AI 诊断；M4 加入操作、审批与审计；M5 再完善全局/上下文对话和诊断知识体验。首页不提前堆放尚未实现的空模块。

## 3. 初步信息架构

顶级导航按产品能力逐步开放：

- 总览：Fleet 健康、重点异常、资源趋势和最近活动。
- 基础设施：机器列表与机器详情。
- 服务：服务拓扑、实例、健康检查和关联资源。
- 事件：告警、事故、确认、静默、诊断、处置与恢复时间线。
- 仓库：GitHub 仓库、提交、Actions、部署版本和关联服务。
- Agent：跨 Fleet 的全局运维对话。
- 操作记录：Runbook、审批、执行结果和审计。
- 设置：接入、通知渠道、数据保留和系统配置。

尚未交付的模块默认隐藏，不展示不可用的空导航入口。页面应支持稳定的深层链接，便于从告警、通知和 Agent 回答直接定位到机器、服务、事件或仓库。

### 机器详情

机器详情逐步形成以下页面或标签：

- 概况：资源、在线状态、版本和关键风险。
- 服务：Docker、systemd、HTTP 检查和关联业务服务。
- 指标：当前值与后续趋势。
- 日志：有限、可审计的取证窗口。
- 终端：后期人工兜底能力。

SSH/终端不作为默认顶级导航。入口应位于具体机器上下文中，明确显示目标机器、连接状态和权限边界；后期可进入独立全屏模式，但始终保留机器身份和审计信息。

### 服务详情

服务详情用于汇总服务实例、健康状态、事件、仓库、部署版本、Runbook 和 Agent 诊断，是后续故障闭环的主要工作空间。

## 4. Agent 对话设计

Agent 对话不应只是一个孤立的聊天页面，而应同时支持全局对话和页面上下文对话。

### 全局 Agent

- 从顶级 Agent 页面询问整个 Fleet。
- 适合跨机器查询、全局风险总结和历史事件检索。
- 回答引用具体机器、服务、事件、日志窗口、仓库文件或 Commit SHA。

### 上下文对话抽屉

- 在主要页面提供可呼出的侧边对话抽屉。
- 机器页面自动携带当前机器上下文。
- 服务页面自动携带服务实例、健康状态、仓库和部署版本。
- 事件页面自动携带事件时间线、证据和已执行操作。
- 仓库页面自动携带当前仓库、分支、提交和关联服务。

用户应能看到当前对话使用了哪些上下文，并可以移除或补充上下文。页面、日志、仓库文本和服务输出始终视为不可信输入，不能改变工具权限或绕过审批。

### 对话与操作边界

- Agent 明确区分事实、推断、建议和证据来源。
- 只读查询可以直接完成并留下记录。
- 重启、部署、回滚等写操作只能生成受控计划，并进入 Runbook、确认、验证和审计流程。
- 对话不能成为自由 Shell 的隐式入口。

### 自然语言操作体验

- 用户表达目标，系统展示识别出的机器、服务、环境、版本、计划步骤、风险、前置检查和验证条件。
- 只读请求在已授予策略内可以直接执行；写请求复用 M4 操作状态机，根据风险进入一次确认或二次确认。
- Runbook 和底层参数由系统根据服务拓扑、Compose/systemd 信息和部署模板生成，用户主要确认业务意图和影响范围，不需要手工编写命令白名单。
- 操作完成后，对话必须返回实际执行结果、健康验证、当前版本和审计记录链接，不能只回复“已完成”。
- 后期临时高风险会话必须明确显示目标机器、权限、有效期和自动撤销时间；它是人工兜底，不是 Agent 的永久自由 Shell。

## 5. 一条命令接入与自动发现

接入体验的完成标准不是“Agent 已注册”，而是用户无需登录 VPS 继续编辑技术配置即可开始监控和诊断：

1. 用户在 Web 中填写机器名称并选择能力档位：仅监控、监控与诊断、完整受控运维。
2. Web 生成一条带本地能力策略的安装命令，并单独展示短期注册令牌，避免令牌进入 Shell 历史。
3. Agent 安装后自动发现 Docker Compose 服务、普通容器、systemd Unit、端口和可能的部署目录，并上报不含敏感目标细节的能力目录。
4. Web 展示候选业务服务、稳定身份和日志能力；当前 Docker 与 systemd 服务均可逐项确认，批量确认后续增加。
5. 用户只补充无法可靠推断的业务信息，例如环境、健康检查和部署方式；仓库从 GitHub App 已授权列表选择，界面不要求填写容器 ID、Unit 参数、`source_key` 或 JSON。

文件日志、任意路径和高风险能力不得因为“自动发现”而默认开放。界面应清楚区分系统发现、用户确认、策略自动授权和拒绝原因。

## 6. 视觉与交互方向

推荐采用精致、克制的现代运维控制台风格：

- 以深色主题为主要方向，同时保证对比度和可访问性。
- 使用统一且有限的正常、注意、异常、离线和进行中状态色。
- 使用紧凑趋势图、事件时间线、服务拓扑和轻量状态动效。
- 重要异常在视觉上优先，但避免大面积持续闪烁和无意义发光。
- 桌面端强调信息密度和并行观察，移动端强调告警查看、确认和关键状态。
- 空状态、加载状态、部分数据缺失和控制平面不可用都应有完整设计。

最终视觉稿应基于真实数据密度与 M2 事件工作流制定，不以简单表格或线框图作为最终界面标准。

## 7. 分阶段实施

### 当前微调

- 建立可扩展的应用外壳、导航和页面宽度规则。
- 统一卡片、指标、状态标签、分组、空状态和错误状态组件。
- 调整首页信息层级，使其具备从机器列表演进为运维总览的空间。
- 保持当前 M1 功能和 API 协议不变，不制作尚未交付能力的假页面。

### M2 完成后

- 使用真实告警、恢复和事件时间线验证首页布局。
- 确定最终视觉语言、仪表盘密度和响应式行为。
- 完成事件中心和上下文 Agent 抽屉的交互原型。

### M3–M5

- M3 增加服务拓扑、仓库、部署版本、有限证据和最小诊断工作区。
- M3 首批事件页已能手动触发只读诊断，并分区展示事实、推断、建议、缺失证据和原始证据编号；后续根据真实事件密度继续调整布局。
- M3 产品化首批已增加 Docker 自动发现、稳定服务身份、证据源目录、安装策略选择和 Web 单服务确认流程，手工 JSON 只保留为兼容入口；批量确认和更多来源后续增强。
- M3 机器失联事件复用同一事件页：页面明确说明诊断只使用控制平面保存的最后心跳、资源与服务快照，不会让用户等待离线 Agent 返回证据。
- M3 GitHub App 首版在首页提供连接状态、授权仓库、当前 HEAD 和手动同步入口；机器详情映射表单从授权仓库列表选择。systemd journal 候选复用现有单服务确认与事件诊断工作区，不新增一套流程。
- M4 增加 Runbook、审批、执行验证和审计界面。
- M5 让全局 Agent 与页面对话共享结构化上下文、证据引用、诊断历史和 M4 操作状态机。

### M6

- 完善 PWA、移动审批、全屏终端、团队协作和自托管产品化体验。

## 8. 设计约束

- Web 只调用控制平面，不直接访问 Agent 或 VPS。
- UI 重构应优先保持现有 API 类型和路由稳定；新增能力通过明确的版本化接口扩展。
- 状态、严重程度、时间和目标资源必须使用一致语义。
- 任何高风险操作都必须在界面中明确展示目标、影响、前置检查、确认步骤和验证结果。
- 视觉效果不得掩盖风险、错误、数据过期或部分失败。
