# M5.3 会话到安全操作计划的显式交接

本文定义 M5.3 的首个范围：在不授予 Provider 工具权限、不改变 Agent 协议、不弱化 M4 独立确认的前提下，把已完成的事件会话显式交接到现有安全重启计划入口。

当前状态：**本地实现与验收完成；模型、迁移、M4 服务抽取、严格 API、Web 交接、自动化测试和真实 PostgreSQL 迁移/约束验证均已通过。尚未生产部署或金丝雀验证。**

2026-07-24 实现审计说明：

- 已按本文边界加入 `0012_m5_conversation_operation_source.py`；Alembic revision ID 使用不超过 `alembic_version VARCHAR(32)` 的 `0012_m5_operation_handoff`。
- 已从现有 M4 入口抽取唯一 `build_restart_plan`，原 `/operations` 与 M5.3 专用入口共用相同预检、活动写互斥和状态转换。
- 已加入候选 GET、严格 restart-plan POST、请求级幂等、来源墓碑、默认关闭开关、同源 Web 代理和事件页显式按钮。
- API 常规回归 164 项通过；隔离 PostgreSQL 16 上 M5.1/M5.2/M5.3 三个门控测试全部通过。Web 46 项、Ruff、ESLint、Web production build、Go 全包和 Compose 配置通过。

M5.1/M5.2.1 继续保持只读：提出问题、生成回答、显示建议或仓库引用都不会创建 Operation。M5.3.1 只有在用户点击独立、明确的计划交接动作后才允许创建一个待确认计划；该点击不等于确认或执行。

## 1. 现有架构审计

### 1.1 可直接复用

- 事件页已经有独立的“创建安全重启计划”按钮，不依赖诊断或会话自然语言。
- `POST /api/v1/operations` 只接受固定 `docker_restart`，拒绝命令、路径、目标容器和其他额外字段。
- M4 会从服务事件派生唯一 `ServiceInstance`，并检查 Agent 在线、Docker 服务、映射、显式 Agent capability、控制平面 `restart_enabled`、非关键服务和观测新鲜度。
- 计划创建后只到 `awaiting_confirmation`；确认端点是独立管理动作，确认后才签名并进入 Agent 可领取队列。
- Agent 只接受 Ed25519 签名、带过期时间和幂等键的 v1/v2 固定任务；执行后仍须新鲜健康观测和稳定窗口。
- 同服务 `active_key` 防止并发写操作；所有状态变化进入 `OperationTransition`。
- Web 写代理已有同源校验，并把管理令牌保留在服务端。
- `Operation` 已能关联来源事件和诊断，并把预检、影响、验证策略和结果冻结为审计记录。

### 1.2 当前缺口

1. `Operation` 没有来源会话轮次，无法审计一次计划是否由哪一轮显式交接。
2. 当前 `OperationPlanCreate` 可接受 `instance_id`；M5.3 会话专用入口不应让浏览器选择实例。
3. 当前计划创建没有调用方请求 ID；按钮重试主要依靠同服务活动操作冲突，不是完整的请求级幂等。
4. 会话 `recommendations` 只有不可信自然语言、风险和引用，没有可信的结构化 `action_type`；不能从文字解析“restart/deploy/rollback”。
5. 事件页现有重启按钮与会话并列，但没有“只创建计划”和“仍需独立确认”的会话交接说明。
6. 现有 `resolve_instance` 主要建立事件/诊断/实例一致性；M5.3 新入口必须额外验证组织、会话、轮次和事件链。
7. 部署需要可信目标 digest；回滚目标必须从失败部署服务端派生。二者都不能从会话文本取得，因此不适合作为首片顺手接入。

结论：**现有 M4 状态机可以复用，M5.3.1 不需要 Provider 工具或 Agent 变化；但必须新增会话来源审计、请求幂等和专用的服务端派生入口。**

## 2. M5.3.1 目标

首个垂直切片严格限定为：

1. 只支持事件作用域会话中的固定 `docker_restart` 计划交接。
2. 只允许从 `completed` 且已保存严格结构化回答的轮次发起。
3. 用户必须点击独立的“准备安全重启计划”按钮；发送问题、轮询回答、查看建议和加载历史都不创建 Operation。
4. 请求体不接受 Agent、实例、服务键、命令、路径、镜像、digest、回滚来源或确认字段。
5. 服务端从 URL 中的事件和轮次派生组织、服务实例及固定动作类型，并复用 M4 预检和活动操作互斥。
6. 同一个客户端请求 ID 重试返回同一个 Operation，不创建重复计划。
7. 新 Operation 最多进入 `awaiting_confirmation`；同一次请求不能确认、签名、排队、领取或执行。
8. 用户随后必须在现有操作详情页查看冻结计划，并通过现有独立确认入口确认。
9. 会话来源进入 Operation 审计，但不把问题、回答、仓库摘录或 Provider 原文复制进 `plan_snapshot`。
10. 保持 M5.1/M5.2.1 只读请求零 Operation 副作用。

## 3. 非目标

- 从“修复它”“执行建议”“部署最新代码”等自然语言自动创建计划。
- 让 Provider 返回工具调用、操作参数、实例 ID、服务键、路径或 digest。
- 在回答生成完成后自动弹出、自动提交或自动确认计划。
- 在同一个 HTTP 请求中创建并确认 Operation。
- 会话内直接执行 Agent Claim/Start/Complete。
- Compose 部署、显式回滚、GitHub branch/commit/push/PR 或其他写操作。
- Shell、任意命令、任意路径、任意容器选择或任意 systemd 写操作。
- 修改 M4 v1/v2 签名协议、状态机、验证窗口或回滚派生规则。
- 全局 Fleet 对话、跨事件计划或多 Agent 自主循环。

## 4. 必须继承的安全边界

- 自然语言建议不是授权；Provider 内容永远是不可信输入。
- 计划创建、计划确认和 Agent 执行是三个不同权限阶段。
- 只有显式服务策略和 Agent capability 同时允许时才能通过预检。
- 首片继续只允许非关键 Docker 服务。
- 确认后任务仍须 Ed25519 签名、过期时间、nonce、幂等键和合法状态转换。
- Web/API 不直接访问 VPS；只有 Agent 出站领取已确认签名任务。
- 执行成功仍取决于后续新鲜 `running`/`healthy` 观测和稳定窗口。
- M5.3 不改变部署双 config-hash 漂移门、同报告 digest/健康验证、显式回滚或禁止回滚链等 M4.2 属性。
- 会话入口不能导入 Agent Claim/Complete 客户端或 GitHub 写客户端。

## 5. 威胁模型

| 威胁 | 后果 | M5.3.1 控制 |
| --- | --- | --- |
| 恶意问题或证据写“立即重启” | 提示注入变成写授权 | Provider 无工具；回答路径零 Operation；只有独立用户 POST 可创建计划 |
| Provider 伪造 `action_type`、实例或按钮 | confused deputy | UI 动作由服务端候选生成；Provider schema不增加操作字段 |
| 用户在请求体注入其他事件/实例/命令 | IDOR 或任意执行 | URL 固定事件/轮次；严格请求体只有请求 ID 和过期秒数；其余字段 `extra=forbid` |
| 跨组织或跨事件轮次 | 错误服务被操作 | 事件、会话、轮次、组织和服务实例在同一事务中连接校验 |
| 双击、刷新或网络重试 | 重复 Operation | 客户端 UUID + 数据库唯一约束 + 幂等读取 |
| 轮次完成后事件映射变化 | 操作旧目标 | 创建计划时重新从当前事件派生实例并运行全部 M4 新鲜预检 |
| 创建计划后自动执行 | 绕过独立确认 | 创建路径上限为 `awaiting_confirmation`；无签名/queued/claim；独立确认端点不合并 |
| 隐藏 CSRF 请求 | 未知计划写入 | Web 代理同源校验、服务端管理令牌、严格 JSON 和大小限制 |
| Critical 服务被会话重启 | 高影响中断 | 复用 M4 `non_critical_service` 与显式 `restart_enabled` 门 |
| 活动部署与会话重启竞争 | 并发写冲突 | 复用同实例 `active_key={instance}:write` 唯一约束 |
| 计划来源文本泄露秘密 | 审计面泄露 | 只保存轮次 ID、回答/context 哈希和固定动作；不复制正文 |
| 会话删除导致 Operation 丢失审计 | 来源断链 | FK `SET NULL`，`plan_snapshot` 保留不含正文的轮次 ID 墓碑和哈希 |
| 通过会话指定部署 digest | 未授权部署 | 首片端点不接受 deploy；部署/回滚仍走独立 M4.2 API |

## 6. 权限与状态边界

```text
用户提问
  -> M5.1/M5.2.1 只读回答（operations 不变）
  -> 用户点击“准备安全重启计划”
  -> M5.3.1 专用 POST
  -> 服务端作用域校验 + M4 预检
  -> Operation.awaiting_confirmation
  -> 用户进入现有操作详情页
  -> 独立点击确认
  -> M4 签名/queued
  -> Agent claim/start/complete
  -> 新鲜健康验证
  -> succeeded/failed
```

强制规则：

- “点击发送问题”和“点击准备计划”不是同一个动作。
- “点击准备计划”和“点击确认执行”不是同一个动作。
- M5.3 POST 不调用 `confirm_operation`，不生成签名，不修改为 `queued`。
- 即使 Provider 建议 `requires_confirmation=false`，首片也忽略该值；M4 写操作仍始终需要确认。
- 候选是否显示不能仅依据 Provider recommendation；它由事件类型和服务端策略决定。

## 7. 最小数据模型

建议新增显式迁移 `0012_m5_conversation_operation_source`，只扩展 `operations`：

| 字段 | 用途 |
| --- | --- |
| `source_conversation_turn_id` | 可空 FK 到 `conversation_turns.id`，`ON DELETE SET NULL`，带索引 |
| `conversation_request_id` | 可空 UUID 字符串；M5.3 计划请求幂等键 |

约束：

- `UNIQUE (organization_id, conversation_request_id)`；现有非会话 Operation 为 `NULL`，不受影响。
- M5.3 创建的 Operation 两个字段都写入；现有 M4 API 不要求它们。
- 轮次删除后 FK 可变为 `NULL`，但 `plan_snapshot.conversation_source` 保留：
  - 当时的 `turn_id`
  - `answer_sha256`
  - `context_manifest_sha256`
  - 固定 `handoff_kind=explicit_user_restart_plan`
- `plan_snapshot` 不保存问题、回答、引用正文、仓库摘录、Provider 原文或用户自由文本。
- `OperationView` 只增加可空 `source_conversation_turn_id`；不向浏览器暴露内部幂等键。

不建议首片新增：

- `ConversationAction`、工具调用或自主任务表。
- Provider 生成的操作参数表。
- 部署目标或回滚目标字段；M4 已有可信字段和派生规则。

## 8. API 协议

### 8.1 只读候选

建议新增：

`GET /api/v1/events/{event_id}/conversation/operation-candidates`

响应由服务端生成：

```json
{
  "event_id": "event-id",
  "candidates": [
    {
      "action_type": "docker_restart",
      "available": true,
      "reason_code": null,
      "impact_summary": "只创建非关键 Docker 服务的待确认重启计划",
      "requires_plan_creation": true,
      "requires_confirmation": true
    }
  ]
}
```

规则：

- GET 不创建 Operation、不运行写预检、不访问 Agent。
- 只做事件/组织/服务映射和静态策略可用性摘要；真正新鲜预检仍在 POST。
- 不返回命令、路径、容器 ID、Agent 凭据或签名字段。
- 不从 Provider recommendation 生成候选。

### 8.2 显式计划交接

建议新增：

`POST /api/v1/events/{event_id}/conversation/turns/{turn_id}/restart-plan`

请求：

```json
{
  "client_request_id": "UUID",
  "expires_in_seconds": 300
}
```

`expires_in_seconds` 与当前 M4 `OperationPlanCreate` 保持一致：默认 `300`，范围 `60..900`。

禁止请求字段：

- `instance_id`
- `agent_id`
- `service_kind`
- `service_key`
- `action_type`
- `command`
- `path`
- `target_digest`
- `rollback_of`
- `confirmed_by`
- `auto_confirm`

处理顺序：

1. 管理认证和组织上下文。
2. 校验事件存在且属于组织。
3. 校验会话属于事件。
4. 校验轮次属于会话/组织且为 `completed`，回答结构有效。
5. 按 `(organization_id, client_request_id)` 查询幂等结果。
6. 服务端从事件派生唯一实例和固定 `docker_restart`。
7. 调用抽取后的 M4 重启计划构建服务，执行现有全部预检和活动写互斥。
8. 保存会话来源 FK、无正文哈希墓碑和转换审计。
9. 返回 `OperationView`；成功状态必须为 `awaiting_confirmation`。

幂等语义：

- 相同请求 ID、事件和轮次重试返回同一 Operation。
- 请求 ID 已用于其他事件/轮次时返回 409，不泄露其他作用域对象。
- 并发首次请求依赖数据库唯一约束收敛到一个 Operation。

### 8.3 现有确认入口不变

继续使用：

`POST /api/v1/operations/{operation_id}/confirm`

M5.3 不增加 `create-and-confirm`、`execute-recommendation` 或自然语言确认端点。确认页必须显示服务、动作、影响、预检、过期时间和会话来源，但不能仅展示 Provider 建议文本。

## 9. 服务层复用

为避免内部 HTTP 调用和安全逻辑分叉，建议从现有 `create_operation` 抽取：

```text
build_restart_plan(
  session,
  instance,
  event,
  diagnostic,
  settings,
  source_metadata
) -> Operation
```

现有 `/operations` 与新 M5.3 专用端点都调用同一服务：

- 同一个 `run_prechecks`
- 同一个 `active_key`
- 同一个 impact/verification policy
- 同一个 planned -> prechecking -> awaiting_confirmation/failed 转换
- 同一个数据库事务和 IntegrityError 冲突处理

禁止：

- M5.3 复制一份较弱的预检。
- M5.3 通过 HTTP 回调自己的 `/operations`。
- 在 conversation.py 中直接构造签名任务。
- 为会话来源改变 M4 transition 合法集合。

## 10. Web 最小交互

- 只在服务事件、服务端候选允许且存在最新 `completed` 轮次时展示交接区。
- 文案固定为“准备安全重启计划”，不使用“立即修复”“执行建议”。
- 按钮旁明确说明：“只创建待确认计划，不会立即访问 Agent 或重启服务。”
- 点击后 POST 严格请求，成功跳转 `/operations/{id}`。
- 操作详情页继续显示现有独立“确认执行”按钮。
- 提交中禁用按钮；网络重试复用同一 `client_request_id`。
- 409 活动操作冲突时提供现有 Operation 链接（若服务端安全返回）。
- Provider Markdown、引用链接、仓库文本不能渲染为操作按钮或绑定点击处理。
- M5.3.1 不在聊天输入框支持 `/restart`、快捷命令或隐式 Enter 执行。

## 11. Provider 与不可信输入规则

- ConversationAnswer schema 首片不新增 `tool_calls`、`action_type` 或操作参数。
- Provider recommendation 继续只是显示文本；`requires_confirmation` 不能降低 M4 要求。
- 用户问题、历史、证据、仓库、诊断、Operation 输出和回答都不参与选择可执行字段。
- 会话来源哈希在服务端从已落库、已脱敏的 answer/context 计算。
- Provider 失败、超时或非法结构的轮次不能发起计划交接。
- 历史已完成轮次不会因为后来 Provider 配置变化而获得新权限。

## 12. 日志与审计

允许记录：

- event/turn/operation ID
- 固定 action type
- 计划创建结果和 M4 reason code
- 幂等命中布尔值
- actor=`local-admin`

禁止记录：

- 用户问题和回答正文
- evidence/repository 摘录
- Provider 原始请求/响应
- 管理令牌、Agent 凭据、签名私钥
- 完整 webhook URL 或 GitHub token

OperationTransition 初始记录建议：

```json
{
  "actor_type": "admin",
  "reason": "conversation restart plan explicitly requested",
  "details": {
    "source": "conversation_handoff",
    "turn_id": "turn-id"
  }
}
```

该记录只证明用户显式请求创建计划，不证明用户已经确认执行。

## 13. 配置与关闭语义

- 新增 `CONVERSATION_OPERATION_HANDOFF_ENABLED=false`，默认关闭；它只控制 M5.3 候选与计划交接，不改变 M5.1/M5.2.1 的只读会话行为。
- 开关关闭时，候选 GET 返回 `200`，其中固定候选为 `available=false`、`reason_code=feature_disabled`；这样 Web 能稳定显示不可用原因，也不会把关闭误判为事件不存在。
- 开关关闭时，显式 restart-plan POST 返回 `409` 和稳定错误码 `conversation_operation_handoff_disabled`，不创建 Operation 或 Transition。
- 开关开启不代表某服务被授权。POST 仍须重新通过当前事件作用域、非关键服务、控制平面 `restart_enabled`、Agent capability、在线/新鲜度和活动操作互斥等全部 M4 预检。
- 配置解析失败应在启动期 fail closed；不得因缺省、非法布尔值或热更新失败自动开启。

## 14. 测试矩阵

### 14.1 零副作用

- 提问、轮询、读取历史、读取候选不会改变 Operation/Transition。
- “修复它”“立即执行”“部署它”等问题只生成回答，不创建计划。
- Provider 返回恶意 action/tool/URL 字段时由现有严格 schema 拒绝。
- M5.3 POST 只新增计划记录和预检转换，不创建签名、任务 nonce 或 Agent Claim。

### 14.2 作用域

- 事件、会话、轮次和组织必须完整一致。
- 其他事件、其他组织、其他会话、pending/running/failed 轮次均不可交接。
- 问题中注入 instance/event/agent ID 不改变服务端派生范围。
- 轮次完成后映射变化时使用当前事件映射并重新预检，不使用历史文本目标。

### 14.3 严格协议与幂等

- 请求只接受 UUID 和过期秒数，所有额外字段拒绝。
- 相同请求 ID 串行/并发重试返回同一 Operation。
- 同请求 ID 用于不同作用域返回 409。
- 双击不会生成两个活动或两个失败计划。

### 14.4 M4 安全复用

- Critical 服务、restart disabled、Agent 无 capability、离线、观测陈旧和非 Docker 均预检失败。
- 已有 deploy/restart 活动操作时返回冲突。
- 成功计划停在 `awaiting_confirmation`，Agent `/next` 返回空。
- 只有另一次独立确认后才产生签名并变为 queued。
- 签名字段、过期、Claim、Start、Complete、验证和取消测试继续通过。

### 14.5 数据与迁移

- 真实 PostgreSQL `0011 -> 0012`、`0012 -> 0011 -> 0012` 和 schema check。
- 已有 Operations 保留且新字段为 NULL。
- 删除会话轮次不会删除 Operation；来源 FK 变 NULL，plan_snapshot 墓碑仍在。
- `(organization_id, conversation_request_id)` 唯一约束并发有效。

### 14.6 Web

- 候选不可用、无完成轮次、机器事件不显示按钮。
- 按钮文案明确“准备计划”且提示不立即执行。
- POST 代理拒绝跨站、超大和额外字段请求。
- 成功跳转现有 Operation 页面；不会自动调用 confirm。
- 加载、409、502 和重复请求状态可控。

### 14.7 回归

- API、Web、Go、Ruff、ESLint、Compose 和 Web production build。
- M2/M3/M4/M5.1/M5.2.1 全量测试。
- 会话只读路径在功能开关关闭时行为和计数不变。

## 15. P0 / P1 / P2

### P0：实现前必须关闭

1. Provider/自然语言不能创建 Operation 或产生可信操作字段。
2. 专用 POST 必须要求显式用户动作、管理认证和同源代理。
3. 事件→会话→轮次→组织→当前服务实例的完整作用域链。
4. 创建与确认严格分离；创建请求绝不能签名、queued 或可领取。
5. 固定 `docker_restart`，请求体不能携带实例、命令、路径、digest 或回滚目标。
6. 复用完整 M4 预检、活动写互斥和状态机，不复制弱化版本。
7. 请求级数据库幂等，双击和并发不生成重复计划。
8. 只读会话和候选读取继续零 Operation 副作用。

### P1：首片完成

1. `0012` 会话来源和请求幂等字段。
2. M4 重启计划服务层抽取与原端点回归。
3. 候选 GET、显式 restart-plan POST 和严格 schema。
4. Web 最新完成轮次交接、说明、加载/错误和跳转。
5. 真实 PostgreSQL、跨组织、并发幂等、Agent 不可领取测试。
6. 文档和生产关闭开关。

### P2：首片稳定后

1. 在会话中展示已有 Operation 的只读时间线和最终验证结果。
2. 从已完成诊断/会话跳转到部署计划页面，但目标 digest 仍由独立可信选择器产生。
3. 显式回滚交接，仅从失败部署 Operation 服务端派生。
4. 多人 RBAC 下区分计划创建者和独立确认者。
5. 操作候选反馈和 runbook 复盘。

## 16. 实施顺序

1. **M5.3a 数据与来源门**：`0012`、Operation 来源轮次、请求幂等和迁移测试。
2. **M5.3b M4 服务层复用**：抽取重启计划构建函数，确保现有端点行为/测试不变。

4. **M5.3d Web**：最新完成轮次的明确计划按钮与现有操作页跳转。
5. **M5.3e 本地验收**：真实 PostgreSQL、并发、零副作用、不可领取和全量回归。
6. **M5.3f 生产计划级金丝雀**：另获授权后部署但保持关闭；开启时只创建并取消一个待确认计划，不确认、不让 Agent 领取。

## 18. 本地验收标准

只有同时满足以下条件才可标记 M5.3.1 本地完成：

- 只读会话路径仍然零 Operation 副作用。
- 恶意问题、证据、仓库或 Provider 输出不能创建计划。
- 只有显式 M5.3 POST 能创建固定重启计划。
- 跨事件/组织/轮次不可创建。
- 请求重试和并发只产生一个 Operation。
- 新计划只到 `awaiting_confirmation`，无签名且 Agent 不可领取。
- 独立确认后仍完整复用 M4 签名和验证闭环。
- M4 原有重启入口行为不变。
- 真实 PostgreSQL 迁移和约束通过。
- M2/M3/M4/M5.1/M5.2.1 与全部自动测试无回归。

## 19. 生产金丝雀边界

本设计不授权部署或生产写操作。未来必须分两级：

### 19.1 计划级金丝雀

- 先备份、preflight、迁移 `0011 -> 0012`、部署和 postflight。
- 首次部署保持 `CONVERSATION_OPERATION_HANDOFF_ENABLED=false`。
- 只在现有非关键、显式 restart-enabled、Agent capability 已授权的 M4 金丝雀服务上开启。
- 提问和读取候选后 Operation 计数不变。
- 用户显式点击一次，新增恰好一个 `awaiting_confirmation` Operation。
- 验证无签名、无 nonce、Agent `/next` 不返回任务。
- 在现有操作页显式取消该计划，记录 canceled 转换。
- 关闭功能开关并清理测试会话；不删除 Operation 审计。

### 19.2 执行级金丝雀

- 必须再次获得用户明确授权，不能从“计划级金丝雀通过”推断。
- 仅使用已通过 M4 生产写验证的金丝雀节点和非关键服务。
- 在操作详情页独立确认，不通过会话自动确认。
- 核对签名、领取、执行、新鲜健康观测、稳定窗口和最终状态。
- 不部署代码、不回滚、不访问任意路径、不扩大 Agent capability。

生产失败时优先关闭单一 M5.3 功能开关；不得回退或弱化 M4 状态机。
