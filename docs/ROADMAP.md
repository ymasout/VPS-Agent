# 实际开发路线图

本文档将原始计划书中的 P0–P6 转换为仓库实际执行的 M0–M6。原始计划书保持不变，本文件负责同步范围、进度和验收结果。

## 1. 状态说明

- `已完成`：代码、文档和当前阶段验证均已完成。
- `进行中`：已开始实现，尚未满足全部验收条件。
- `待开始`：范围已确认，尚未进入实现。
- `冻结`：明确不属于当前产品路线。

## 2. 总体进度

| 实际里程碑 | 对应计划 | 状态 | 核心目标 |
| --- | --- | --- | --- |
| M0：项目骨架 | P0 的工程基础 | **已完成** | Monorepo、技术栈、配置、日志、测试和开发环境 |
| M1：机器可见 | P1：可见 | **已完成** | Agent 安全接入、资源采集、服务状态和真实 VPS 页面 |
| M2：异常可通知 | P2：可通知 | **已完成** | 告警状态机、钉钉机器人、去重和恢复通知 |
| M3：上下文与 AI 诊断 | P3 + P5 核心闭环 | **进行中** | 服务/目录/日志/仓库映射、有限取证和结构化诊断 |
| M4：操作可处置 | P4：可处置 | 待开始 | 安全重启、Runbook、审批、验证和审计 |
| M5：诊断体验增强 | P5 完整体验 | 待开始 | Agent 对话、仓库知识、更多证据源和诊断工作空间 |
| M6：自托管产品化 | P6 | 待开始 | PWA、更多模板、升级备份、协作与开源分发 |
| SaaS/商业化 | 计划第 7 章 | **冻结** | 满足原计划前置条件前不启动 |

## 4. M1：机器可见

状态：**已完成**

### 范围

1. Agent 一次性令牌注册与绑定。
2. Agent 身份凭证、吊销和后续轮换空间。
3. 心跳持久化与在线/离线判断。
4. VPS 主机名、系统、架构、Agent 版本和能力信息。
5. CPU、内存和磁盘快照。
6. Docker 容器状态。
7. systemd 服务状态。
8. HTTP 健康检查。
9. Web Fleet 首页真实 VPS 总览。
10. VPS 详情页资源和服务状态。

### 默认频率

| 项目 | 默认值 |
| --- | ---: |
| 心跳 | 30 秒 |
| CPU、内存、磁盘 | 60 秒 |
| Docker/systemd | 30–60 秒 |
| HTTP 健康检查 | 30–60 秒 |

### 明确不包含

- 告警通知和 Telegram。
- GitHub 仓库关联。
- 服务重启、部署和自由 Shell。
- AI 诊断和知识库。
- 长期指标趋势和独立时序数据库。

### 实际结果

- Fleet 已有 4 条真实机器记录，其中 3 台外部 VPS 持续上报，满足原计划“至少 3 台”验收线。
- 控制平面宿主机已验证离线/在线判断，并保留原身份升级到 `v0.2.4`；Fleet 未产生重复机器。
- 4 台真实 VPS 当前全部在线，Agent 版本统一为 `v0.2.4`。
- 注册、认证、基础资源、Docker/systemd、HTTP 健康检查、首页和详情页均已在真实环境验证。
- 国内网络不再依赖目标 VPS 直连 GitHub Release CDN。

### 后续质量工作

- [x] 增加 Go 客户端、身份文件和更多采集/配置边界测试。
- [x] 优化安装页，固定机器名称、分离安装命令与一次性令牌，并处理控制平面不可用场景。

2026-07-14 已完成首轮质量加固：Web 与 API 各 16 项测试通过，全部 Go 包测试通过，Web 生产构建、ESLint、Ruff 和 `go vet` 通过。真实 PostgreSQL 集成、安装器沙箱和长时间稳定性测试继续纳入常规质量演进，不阻塞 M2。

### 验收条件

- 至少 3 台真实或测试 VPS 稳定展示资源与服务状态。
- Agent 注册令牌短期有效、一次性使用且不明文保存。
- Agent 重启和短暂断线后恢复同一身份。
- 无效身份无法提交心跳或监控数据。
- 控制台正确展示在线、离线和最后心跳时间。
- CPU、内存、磁盘数据与目标 VPS 实际情况一致。
- Docker/systemd 状态能够区分运行、停止和失败。
- 首页用于总览，详情页承载资源和服务明细。
- 异常或不完整上报不会导致 API 或页面崩溃。

## 5. M2：异常可通知

状态：**已完成**

- [x] 服务异常形成具有活动指纹的可去重事件。
- [x] 首个通知通道调整为钉钉自定义机器人，支持 Webhook 加签和 Markdown 消息。
- [x] 支持 Pending、Firing、Acknowledged、Silenced、Resolved 状态与管理 API。
- [x] 服务恢复时只生成一次恢复通知投递。
- [x] 通知投递持久化，并与 Agent 报告响应解耦。
- [x] 在真实 PostgreSQL 与钉钉群完成异常、去重和恢复验收。
- [x] 使用真实事件密度验证异常、重复异常与恢复的状态转换和通知次数。

### 实际结果

- 2026-07-16 已在生产控制平面完成部署和金丝雀杀手路径演练。
- 异常连续两次观测后进入 Firing；后续 9 次重复异常没有产生新投递。
- 服务恢复后进入 Resolved；投递序号按 Firing `sequence=1`、Resolved `sequence=2` 递增。
- 生产钉钉群只收到一张异常卡和一张恢复卡，两条投递均发送成功。

### 转入后续路线

- Agent 失联与恢复事件转入 M3，作为诊断触发源和机器级证据。
- 事件诊断工作空间转入 M3；安全操作代理与审批转入 M4。
- Agent 全局/上下文对话和更完整的仓库知识体验转入 M5。
- 独立通知重试调度、Docker 退出码语义和并发事件冲突继续作为跨里程碑质量工作，不阻塞 M2 完成。

Telegram 不再作为首个通知通道，但保留在后续多通道适配范围内。

## 6. M3：上下文与 AI 诊断

状态：**进行中**

- [x] 建立 VPS、业务服务、服务实例、部署目录、日志来源、仓库和部署版本的首批结构化映射。
- [x] 通过 Agent 本地显式白名单收集 Docker 有限日志窗口，并组合服务状态、资源快照和版本证据。
- [x] 对首批证据设置时间、行数、字节数和超时限制，并在 Agent 上传前和控制平面存储前双重脱敏。
- 使用 GitHub App 最小权限授权和 Webhook 验签；只读取授权仓库和白名单文件，并保留 Commit SHA。
- [x] 诊断输出固定区分事实、推断、建议、置信度和证据来源；原始观察证据与诊断结论分开保存。
- [x] 诊断提供者仅进行只读分析，没有命令、服务修改或审批绕过入口。
- [x] 提供事件诊断 API 和最小诊断工作区。
- [x] 在隔离的真实 Caddy、PostgreSQL、Agent 和 Docker 服务上完成首条端到端闭环验收。
- 增加 Agent 失联与恢复事件，作为机器级诊断触发源。

### 完成定义

- 至少一个真实服务能映射到明确目录、受限日志来源和 GitHub 仓库版本。
- 一次真实异常能形成有来源、有限且已脱敏的证据包。
- AI 诊断能够引用证据输出事实、推断和建议，模型失败或证据缺失时返回受控结果。
- 诊断链路没有任意 Shell、任意路径读取或自动写操作入口。

## 10. 路线变更规则

- 新需求先归入对应里程碑，不直接扩大当前里程碑。
- 当前里程碑的新增范围必须同时补充验收条件。
- 延后能力保留在路线图，不从原计划中删除。
- 每次里程碑完成后同步状态、实际结果、偏差和遗留问题。
- SaaS 保持冻结，除非原计划规定的安全、产品和运营前置条件全部满足。
