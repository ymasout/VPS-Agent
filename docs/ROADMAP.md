### 2.1 终局产品体验

项目最终应把多台 VPS 运维收敛为以下体验，而不是让用户长期维护容器 ID、`source_key`、JSON 白名单或逐条命令：

1. 用户在控制台选择机器名称和能力档位，复制一条安装命令完成 Agent 安装、注册和策略绑定。
2. Agent 自动发现 Docker Compose 服务、普通 Docker 容器、systemd Unit、端口和可识别的部署目录，使用稳定身份生成候选业务服务与证据源。
3. 控制台展示发现结果和推荐映射；低风险默认项可以按预先选择的策略自动接受，其余只需在 Web 中确认，不要求登录 VPS 编辑配置。
4. 用户通过全局或页面上下文对话提出“检查异常”“部署指定版本”“回滚上一版本”等目标；系统先生成可解释计划，再由权限策略决定自动执行、一次确认或二次确认。
5. 执行只通过有类型、可校验、带有效期和签名的能力或 Runbook 完成，并在操作后进行健康验证、记录版本和生成审计事件。
6. 高风险通用命令只作为后期临时授权的兜底能力：限定机器、身份、时间和影响范围，默认不可用，自动过期，并保留命令与结果审计。

权限边界必须由系统强制执行，不能依赖模型“主动回避”敏感信息：

| 能力层级 | 默认交互 | 典型范围 |
| --- | --- | --- |
| 只读观察 | 可按已授予策略自动执行 | 状态、指标、有限日志、版本、仓库只读信息 |
| 受控写操作 | 展示计划并按策略确认 | 重启、拉取、部署、回滚、清理和健康验证 |
| 高风险临时会话 | 明确授权、二次确认、限时和完整审计 | 非模板化命令、系统配置修改和人工终端兜底 |

GitHub 凭据保留在控制平面。拉取和部署由 Agent 执行签名任务；修改或推送代码由控制平面通过用户明确授权的 GitHub App 创建分支、提交或 PR，再把已批准的 Commit 或镜像交给 M4 部署流程，不能把长期仓库写凭据下发到 VPS。

## 10. 路线变更规则

- 新需求先归入对应里程碑，不直接扩大当前里程碑。
- 当前里程碑的新增范围必须同时补充验收条件。
- 延后能力保留在路线图，不从原计划中删除。
- 每次里程碑完成后同步状态、实际结果、偏差和遗留问题。
- SaaS 保持冻结，除非原计划规定的安全、产品和运营前置条件全部满足。
