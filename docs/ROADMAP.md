# 实际开发路线图

本文档将原始计划书中的 P0–P6 转换为仓库实际执行的 M0–M6。原始计划书保持不变，本文件负责同步范围、进度和验收结果。

## 1. 状态说明

- `已完成`：代码、文档和当前阶段验证均已完成。
- `进行中`：已开始实现，尚未满足全部验收条件。
- `待开始`：范围已确认，尚未进入实现。
- `冻结`：明确不属于当前产品路线。

## 2. 总体进度

| 实际里程碑 | 对应计划 | 状态 | 核心目标 |
| --- | --- | --- | --- |
| M0：项目骨架 | P0 的工程基础 | **已完成** | Monorepo、技术栈、配置、日志、测试和开发环境 |
| M1：机器可见 | P1：可见 | **进行中** | Agent 安全接入、资源采集、服务状态和真实 VPS 页面 |
| M2：异常可通知 | P2：可通知 | 待开始 | 告警状态机、Telegram、去重和恢复通知 |
| M3：服务可关联 | P3：可关联 | 待开始 | 服务拓扑、GitHub App、仓库和部署版本关联 |
| M4：操作可处置 | P4：可处置 | 待开始 | 安全重启、Runbook、审批、验证和审计 |
| M5：问题可诊断 | P5：可诊断 | 待开始 | 日志取证、AI 诊断、证据引用和修复计划 |
| M6：自托管产品化 | P6 | 待开始 | PWA、更多模板、升级备份、协作与开源分发 |
| SaaS/商业化 | 计划第 7 章 | **冻结** | 满足原计划前置条件前不启动 |

完成定义已满足。完整记录见 [PROJECT_STATUS.md](./PROJECT_STATUS.md)。

## 4. M1：机器可见

状态：**进行中**

### 范围

1. Agent 一次性令牌注册与绑定。
2. Agent 身份凭证、吊销和后续轮换空间。
3. 心跳持久化与在线/离线判断。
4. VPS 主机名、系统、架构、Agent 版本和能力信息。
5. CPU、内存和磁盘快照。
6. Docker 容器状态。
7. systemd 服务状态。
8. HTTP 健康检查。
9. Web Fleet 首页真实 VPS 总览。
10. VPS 详情页资源和服务状态。

### 默认频率

| 项目 | 默认值 |
| --- | ---: |
| 心跳 | 30 秒 |
| CPU、内存、磁盘 | 60 秒 |
| Docker/systemd | 30–60 秒 |
| HTTP 健康检查 | 30–60 秒 |

### 明确不包含

- 告警通知和 Telegram。
- GitHub 仓库关联。
- 服务重启、部署和自由 Shell。
- AI 诊断和知识库。
- 长期指标趋势和独立时序数据库。

### 当前实现进度

- [x] PostgreSQL 数据模型和 M1 迁移基线。
- [x] 一次性注册令牌与 Agent 独立凭证。
- [x] Bearer 认证上报、心跳持久化和在线状态计算。
- [x] Linux CPU、内存和根磁盘采集。
- [x] Docker、systemd 和 HTTP 健康检查采集器。
- [x] Fleet 首页真实数据展示。
- [x] VPS 详情页资源与服务展示。
- [x] 单个 Compose Agent 端到端注册、上报和页面验证。
- [x] 首台真实 VPS 注册、systemd 托管、资源上报和在线/离线验证。
- [x] 服务状态概览、异常优先、分类折叠和 ACTIVE/SUB 语义修正。
- [x] systemd 状态解析单元测试。
- [x] GitHub Release 多架构构建、SHA-256 校验和一键安装/升级脚本。
- [x] Fleet 首页安全生成一次性注册令牌和安装命令。
- [x] 国内网络可用的控制平面 Release 下载中转与服务状态兼容分类。
- [x] 首个正式 `v0.2.2` Release、产物校验和版本验证。
- [ ] 使用 Release 安装器接入剩余 2 台 VPS。
- [ ] 3 台真实或测试 VPS 稳定运行验收。
- [ ] 修正版本发布后的真实 systemd 运行/停止/失败状态复核。
- [x] 断线超时和离线状态验收。
- [ ] Go 客户端和更多采集边界测试。

### 验收条件

- 至少 3 台真实或测试 VPS 稳定展示资源与服务状态。
- Agent 注册令牌短期有效、一次性使用且不明文保存。
- Agent 重启和短暂断线后恢复同一身份。
- 无效身份无法提交心跳或监控数据。
- 控制台正确展示在线、离线和最后心跳时间。
- CPU、内存、磁盘数据与目标 VPS 实际情况一致。
- Docker/systemd 状态能够区分运行、停止和失败。
- 首页用于总览，详情页承载资源和服务明细。
- 异常或不完整上报不会导致 API 或页面崩溃。

## 5. M2：异常可通知

状态：待开始

- 服务异常形成可去重的有状态事件。
- 首个通知通道使用 Telegram。
- 支持 Pending、Firing、Acknowledged/Silenced、Resolved 状态。
- 服务恢复时只发送一次恢复通知。

## 6. M3：服务可关联

状态：待开始

- 建立 VPS、服务、服务绑定、仓库和部署版本拓扑。
- 使用 GitHub App 最小权限授权和 Webhook 验签。
- 只同步白名单文档并保留 Commit SHA。

## 7. M4：操作可处置

状态：待开始

- 首先实现用户确认后的单服务安全重启。
- 所有操作经过 Runbook 白名单、参数校验和健康验证。
- 后续逐步增加部署指定版本、重新部署和回滚。

## 8. M5：问题可诊断

状态：待开始

- 收集有限日志窗口、资源、服务状态、版本和仓库证据。
- AI 输出固定区分事实、推断、建议和证据来源。
- AI 不直接执行命令，也不能绕过审批。

## 9. M6：自托管产品化

状态：待开始

- 安装、升级、备份、恢复和审计可验证。
- 增加 PWA、移动审批、更多模板和通知通道。
- Web SSH 和实时终端作为后期人工兜底能力。
- 单人自用成熟后再评估团队协作和开源分发。

## 10. 路线变更规则

- 新需求先归入对应里程碑，不直接扩大当前里程碑。
- 当前里程碑的新增范围必须同时补充验收条件。
- 延后能力保留在路线图，不从原计划中删除。
- 每次里程碑完成后同步状态、实际结果、偏差和遗留问题。
- SaaS 保持冻结，除非原计划规定的安全、产品和运营前置条件全部满足。
