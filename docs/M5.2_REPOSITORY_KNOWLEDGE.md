# M5.2 GitHub 白名单仓库知识检索

本文定义 M5.2 的首个范围：在不扩大 GitHub App、Agent 或 M4 权限的前提下，把控制平面已经同步、脱敏并限制大小的 GitHub 白名单文件快照接入 M5.1 事件会话。

当前状态：**M5.2.1 已完成本地实现和验收，已提交推送 `54f356d`；生产只读金丝雀于 2026-07-24 通过（功能默认关闭）。**

M5.1 已于 2026-07-23 完成 deterministic 与真实 HTTP JSON Provider 两轮生产只读金丝雀；当前生产已恢复并保持 `deterministic` 已知良好状态。M5.2 不重新定义会话权限，不增加工具调用，不进行 GitHub 写操作。

## 1. 审计结论

### 1.1 与现有架构兼容

M5.2 可以在现有模型上实现，不需要整仓同步、向量数据库或新的 GitHub 权限：

- `Repository` 保存组织作用域内的仓库身份。
- `GitHubRepositoryBinding` 保存 GitHub App 安装授权、启用状态、当前 `head_sha`、同步时间和错误。
- `GitHubRepositoryFile` 已保存精确白名单路径在指定 Commit 下的有界脱敏快照、内容摘要和截断状态。
- 每次 GitHub 同步只申请安装令牌的 `contents:read`，白名单最多 16 个路径，单文件最多 64 KiB。
- 安装暂停、删除或仓库撤权会禁用 Binding，并删除文件快照。
- `ServiceInstance -> DeploymentVersion -> Repository` 已能把服务实例连接到仓库和已记录 Commit。
- M3 诊断已能把仓库文件快照转换为 `repository_file` 类型的 `EvidenceItem`；M5.2 应复用其授权、脱敏和 Commit 语义，但不要求先创建新的诊断。
- M5.1 已有事件/组织作用域、总上下文预算、严格 Provider schema、不透明引用 ID、落库前引用二次校验和零 Operation 副作用测试。

### 1.2 必须正视的现有边界

1. `GitHubRepositoryFile` 只保存最近一次同步的当前 HEAD 快照。同步会先删除旧文件行，因此当前不能检索历史 Commit 的文件正文。
2. `DeploymentVersion.commit_sha` 可能为空，也可能与 GitHub Binding 的 `head_sha` 不同。当前 HEAD 不能自动当作正在运行的代码。
3. `GitHubRepositoryBinding` 自身没有 `organization_id`；任何查询必须通过 `Repository.organization_id` 连接限定组织。
4. 同步失败可能留下 `last_error`；M5.2 不能把不完整或失败快照伪装成完整知识。
5. 撤权会物理删除文件快照。会话引用不能使用阻止撤权清理的 `RESTRICT` 外键。
6. 仓库 README、Compose 或其他白名单文件全部是不可信数据，可能包含提示注入、伪造指令、密钥、超长行、二进制替代文本或误导性的运维建议。
7. 当前事件会话只接受事件 ID；用户问题中的仓库名、服务 ID、Commit 或路径不能成为可信数据库选择器。

结论：**M5.2 没有架构阻塞，但提交检索结果前必须关闭作用域、撤权、Commit 混淆、引用墓碑和上下文预算五类安全门。**

## 2. M5.2.1 目标

首个垂直切片严格限定为：

1. 只在已有服务事件会话中启用仓库知识；不新增全局 Fleet 或任意仓库聊天。
2. 从事件服务端派生唯一 `ServiceInstance`、最新 `DeploymentVersion`、关联 `Repository` 和已启用 GitHub Binding。
3. 只检索数据库中已有的 `GitHubRepositoryFile` 白名单脱敏快照；会话请求不访问 GitHub 网络，也不触发同步。
4. 从有界问题中确定性提取标识符、配置键、路径片段或明确短语，并返回少量匹配摘录。
5. 每个仓库来源固定显示仓库、路径、快照 Commit、已部署 Commit 关系、同步时间和截断状态。
6. 复用 M5.1 的结构化回答、Provider 生命周期、上下文预算、引用清单、二次验证、历史和 Web 展示。
7. 仓库撤权或快照被替换后，不再提供正文；历史引用显示为不可用墓碑，但仍保留当时的仓库/路径/Commit/摘要审计信息。
8. 继续证明提问不会创建或修改 Operation、OperationTransition、EvidenceRequest、GitHub Binding/File 或 Agent 任务。

## 3. 非目标

- 整仓 clone、Git tree 递归抓取或任意路径读取。
- 动态扩大 `GITHUB_ALLOWED_FILE_PATHS`。
- 向量数据库、Embedding、语义搜索、复杂 RAG 或多 Agent 检索循环。
- 在会话请求中实时调用 GitHub、自动同步或等待 Webhook。
- 任意仓库选择器、跨服务搜索、跨事件搜索或全局代码问答。
- GitHub branch、commit、push、PR、Issue 或其他写操作。
- 把仓库文件中的命令、工作流、Prompt 或“执行建议”解释成授权。
- 把 GitHub HEAD 自动解释为已部署版本。
- 修改 Agent、M4 v1/v2 协议、签名、部署或回滚状态机。
- 保存 GitHub 安装令牌、App 私钥、Webhook Secret 或未脱敏原文。

## 4. 必须继承的安全边界

- GitHub App 权限继续只有现有安装范围和 `contents:read`。
- GitHub 凭据只存在于控制平面配置和短期内存，不进入数据库、浏览器、Agent、会话上下文或日志。
- M5.2 Provider 仍无工具接口；检索模块不能导入 GitHub 写客户端或 M4 写服务。
- Web/API 不直接访问 VPS 或 Agent。
- 用户问题只用于在服务端已派生的文件集合内生成匹配词，不参与选择组织、事件、服务、仓库、Commit 或文件路径。
- 所有 SQL 使用参数化表达式；不得把用户文本拼接为 SQL、正则、路径、URL 或 GitHub API 路径。
- 仓库内容与用户问题一样是不可信数据；系统指令和权限边界不从仓库正文生成。
- 自然语言中的“修改代码”“提交修复”“部署它”仍只是未授权意图，不产生 GitHub 写入或 M4 Operation。

## 5. 威胁模型

| 威胁 | 后果 | M5.2.1 控制 |
| --- | --- | --- |
| 用户在问题中指定其他仓库/事件/组织 ID | IDOR 或跨作用域读取 | 仓库只能从当前事件的受作用域服务实例和部署版本服务端派生；忽略问题中的选择器 |
| 已撤权或暂停的 GitHub 安装仍可检索 | 撤权后数据残留 | 查询时要求 Binding `enabled=true`；撤权删除快照；落库前再次校验；历史引用仅保留无正文墓碑 |
| 同步期间快照被替换 | 引用已不是 Provider 看过的内容 | 上下文记录文件 ID、Commit、内容 SHA；回答落库前重新校验三者和 Binding |
| GitHub HEAD 与部署 Commit 不同 | 把未部署代码当成运行事实 | 计算 `aligned/mismatch/unknown`；所有引用显示 Commit；只有 `aligned` 仓库引用可进入“已确认事实” |
| 同步部分失败或过旧 | 使用不完整、陈旧知识 | `last_error` 时默认不提供正文；记录同步时间和 stale 标记；回答必须暴露缺失信息 |
| 仓库正文提示注入 | 诱导工具调用、伪造事实或提升权限 | 正文标记为不可信；Provider 无工具；严格 schema/引用校验；仓库内容不能生成选择器或授权 |
| 仓库正文包含密钥 | Provider、日志或 UI 泄漏 | GitHub 同步时脱敏，进入会话前再次脱敏；日志不记录问题、匹配词、摘录或响应正文 |
| 通配符、正则或超长查询 | SQL 注入、ReDoS、内存/CPU 放大 | 不接受客户端正则；确定性词法提取；限制字符、词数、词长、文件数、摘录数和总字节 |
| 二进制或异常 Unicode | 错误匹配、截断异常 | 复用 UTF-8 replacement/NFKC 规范化；按 UTF-8 字节截断；不执行或解析文件 |
| 仓库上下文挤占事件证据 | 降低诊断质量 | 仓库独立子预算；事件/服务/诊断证据优先于仓库，历史仍最低 |
| Provider 伪造仓库引用 | 虚假 Commit 或路径 | Provider 只回传本轮不透明引用 ID；仓库/路径/Commit 由服务端显示 |
| 仓库引用外链被注入 | 跳转到恶意域名 | 链接只由服务端基于固定 GitHub 主机和已校验 owner/repo/path/Commit 构造；Provider 不返回 URL |
| 会话成为 GitHub/M4 confused deputy | 代码写入或生产操作 | 无写客户端、无工具、无 Operation 依赖；数据库计数和 GitHub 写请求零副作用测试 |

## 6. 作用域派生

检索范围只能按下列服务端链路产生：

```text
AlertEvent(local)
  -> ServiceInstance(agent_id, service_kind, service_key)
  -> latest DeploymentVersion(instance_id)
  -> Repository(organization_id=local)
  -> GitHubRepositoryBinding(enabled=true)
  -> GitHubRepositoryFile(repository_id, commit_sha=head_sha)
```

规则：

1. `source=agent` 的机器事件没有服务仓库范围，返回“当前事件无仓库上下文”，不回退到 Fleet 全局仓库。
2. 服务事件必须唯一解析一个当前组织的 ServiceInstance；零个或多个匹配均不检索。
3. 使用按 `recorded_at DESC, id` 确定排序的最新 DeploymentVersion。
4. DeploymentVersion 没有 Repository 时不检索。
5. Repository 必须属于当前组织；Binding 必须启用且对应当前配置安装。
6. Binding 有 `last_error` 时首版 fail closed：只返回状态元数据，不把现有文件正文放入 Provider。
7. 文件必须满足 `repository_id`、`commit_sha == binding.head_sha`；不接受用户提交的路径或 Commit。
8. 问题中的仓库名、文件名和 Commit 只能作为普通匹配文本，不能替换上述链路。

## 7. Commit 语义

每次检索计算：

- `aligned`：`deployment_commit_sha == repository_commit_sha`。
- `mismatch`：两者都存在且不同。
- `unknown`：部署 Commit 为空。

强制规则：

- 所有仓库引用显示完整 Commit（UI 可缩写显示，但 API 保留完整值）。
- `aligned` 内容可以支持事实、推断和建议。
- `mismatch` 或 `unknown` 内容首版只能支持推断或建议，不能作为 `facts` 的引用；违反时整轮以固定错误码受控失败。
- `mismatch` 回答必须说明“当前仓库 HEAD 不等于已记录部署 Commit”；不得声称该内容正在 VPS 上运行。
- 当前模型没有历史 Commit 正文时，不实时向 GitHub补取。回答将其列为缺失证据。
- 后续若要保存多 Commit 历史，必须另行设计保留上限、撤权清理和引用墓碑，不在 M5.2.1 顺手扩大。

## 8. 确定性检索规则

### 8.1 查询词提取

- 问题沿用 M5.1 的 2000 字符/8 KiB 上限。
- 使用 NFKC、Unicode 小写和空白规范化。
- 只提取：
  - 反引号或引号中的 2–64 字符明确短语；
  - 2–64 字符 ASCII 标识符、配置键和路径片段；
  - 2–16 字符连续中文短语。
- 删除重复项和固定停用词；最多 8 个词。
- 不接受正则表达式或通配符语义；`%`、`_`、反斜杠等只按普通字符处理。
- 没有有效查询词时不扫描正文，返回缺失信息，提示用户提供配置键、错误标识符、文件名或明确短语。

### 8.2 匹配与摘录

- 首版只扫描作用域内最多 16 个现有白名单文件，每个文件本身已不超过 64 KiB。
- 同时匹配服务端已有的文件路径和脱敏正文。
- 不执行 YAML、Markdown、Compose、代码或模板；全部按纯文本处理。
- 每个文件最多 2 个摘录，每个摘录最多 2 KiB；摘录按完整 UTF-8 边界和行边界截取。
- 单轮最多 8 个仓库上下文项，仓库正文总预算默认 24 KiB。
- 排序固定为：
  1. Commit `aligned` 优先；
  2. 路径精确/前缀匹配优先；
  3. 命中不同查询词数量；
  4. 命中次数；
  5. 路径、文件 ID 确定排序。
- 不把“没有命中”解释为仓库中不存在；只说明当前白名单和查询词未找到。

### 8.3 会话总预算

继续服从 M5.1 的默认 128 KiB 总预算，建议优先级：

1. 当前事件与 Agent/服务摘要。
2. 最新诊断的已引用证据。
3. Commit `aligned` 的仓库摘录。
4. 其他诊断证据。
5. `mismatch/unknown` 仓库摘录。
6. Operation 只读摘要。
7. 对话历史。

仓库 24 KiB 是子上限，不是额外预算；不得把总请求扩大到 128 KiB 之外。

## 9. 最小数据模型

### 9.1 复用现有表

M5.2.1 不新增仓库索引、Chunk、Embedding 或检索历史表，复用：

- `repositories`
- `github_repository_bindings`
- `github_repository_files`
- `deployment_versions`
- M5.1 `conversation_sessions` / `conversation_turns` / `conversation_citations`

### 9.2 `0011_m5_repository_citations`

实际只扩展 `conversation_citations`：

| 字段 | 用途 |
| --- | --- |
| `repository_file_id` | 可空 FK 到 `github_repository_files.id`，`ON DELETE SET NULL` |
| `repository_full_name` | 生成回答时的仓库名墓碑，最大 255 |
| `repository_path` | 生成回答时的白名单路径墓碑，最大 512 |
| `repository_commit_sha` | Provider 实际看到的快照 Commit |
| `repository_deployment_commit_sha` | 本轮已记录的部署 Commit，可空 |
| `repository_deployment_relation` | `aligned/mismatch/unknown`，数据库 CHECK 限定 |
| `repository_synchronized_at` | 生成回答时的 Binding 同步时间 |
| `repository_truncated` | 生成回答时的文件截断状态 |
| `repository_stale` | 生成回答时的快照陈旧状态 |

同时：

- `source_type` CHECK 增加 `repository_file`。
- 来源目标 CHECK 对 `repository_file` 要求仓库/路径/Commit/关系元数据完整；`repository_file_id` 允许因后续同步或撤权变为空。
- 其他来源仍保持恰好一个现有目标 FK。
- 继续保存已有 `snapshot_sha256`、`source_collected_at` 和 `source_label`。
- 不保存摘录正文，防止撤权后通过历史会话继续读取仓库内容。

为何不用 `RESTRICT`：现有 GitHub 同步和撤权必须删除旧快照。`RESTRICT` 会让一条历史会话阻止权限撤销；`SET NULL + 不含正文的不可变元数据` 才能同时保持审计和撤权。

### 9.3 不新增搜索表

首版数据量上限为单仓库最多 16 × 64 KiB，且只检索单个事件派生仓库。直接在应用内对已取出的有界文本做确定性匹配即可。只有真实规模和性能证据证明需要时，才评估 PostgreSQL FTS；向量数据库不作为默认下一步。

## 10. 上下文与引用扩展

新增内部 `RepositoryKnowledgeItem`，至少包含：

- 本轮不透明 `citation_id`
- `repository_file_id`
- `repository_id`
- `full_name`
- `path`
- `repository_commit_sha`
- `deployment_commit_sha`
- `deployment_relation`
- `content_sha256`
- `fetched_at` / `synchronized_at`
- `truncated` / `stale`
- 已二次脱敏的有界摘录
- 命中类型和数量（只进入 manifest，不写日志）

Provider 只看到：

- 不透明 citation ID；
- 服务端生成的来源标签；
- Commit 关系；
- 标记为 `untrusted_repository_excerpt` 的摘录。

Provider 不看到 GitHub token、安装 ID、GitHub Repository 数字 ID、内部数据库选择器或任意外部 URL。

回答落库前逐项重新验证：

1. 当前事件、ServiceInstance、DeploymentVersion 和 Repository 作用域未改变到另一个组织/服务。
2. Binding 仍启用且无同步错误。
3. 文件行仍存在，Commit 和内容 SHA 与本轮上下文一致。
4. `facts` 中的仓库引用全部为 `aligned`。
5. 引用通过后才创建 `ConversationCitation`；失败使用固定 `repository_citation_scope_invalid`，不部分保存回答。

## 11. API 协议

### 11.1 提问接口不变

继续使用：

`POST /api/v1/events/{event_id}/conversation/turns`

请求体不增加 `repository_id`、`service_id`、`commit_sha`、`path`、`tool` 或过滤器。仓库作用域完全由事件派生。

当功能开关启用且问题包含有效检索词时，服务端在调用 Provider 前追加仓库上下文。幂等、单活动轮次、Provider 失败和轮询协议保持 M5.1 语义。

### 11.2 会话读取扩展

现有 ConversationCitationView 对 `repository_file` 增加只读元数据：

```json
{
  "id": "ctx_...",
  "source_type": "repository_file",
  "source_id": "snapshot-id-or-null",
  "source_label": "GitHub owner/repo · compose.yaml @ abc1234",
  "source_collected_at": "2026-07-24T00:00:00Z",
  "href": "server-generated-fixed-github-url-or-null",
  "repository": {
    "full_name": "owner/repo",
    "path": "compose.yaml",
    "commit_sha": "full-sha",
    "deployment_commit_sha": "full-sha-or-null",
    "deployment_relation": "aligned",
    "available": true
  }
}
```

规则：

- `source_id` 在快照撤权/替换后可以为 `null`。
- `available=false` 时不返回文件正文链接；UI 显示历史引用墓碑。
- `href` 只能由服务端构造到固定允许的 GitHub 主机和精确 Commit；任何字段校验失败时为 `null`。
- Provider 返回的 URL 或 Markdown 链接永不直接进入 `href`。

### 11.3 首版不新增公开搜索端点

M5.2.1 不提供“任意关键词 + 任意仓库”的独立搜索 API。检索只作为事件会话内部步骤，避免先制造一个更宽的仓库读取面。若后续服务详情页确实需要独立搜索，再以服务 ID 为受作用域父资源单独设计。

## 12. 配置

建议新增保守配置：

- `CONVERSATION_REPOSITORY_KNOWLEDGE_ENABLED=false`
- `CONVERSATION_REPOSITORY_MAX_CONTEXT_BYTES=24576`
- `CONVERSATION_REPOSITORY_MAX_RESULTS=8`
- `CONVERSATION_REPOSITORY_MAX_TERMS=8`
- `CONVERSATION_REPOSITORY_MAX_EXCERPT_BYTES=2048`
- `CONVERSATION_REPOSITORY_STALE_SECONDS=86400`

约束：

- 功能默认关闭，代码部署与生产启用分开。
- 所有上限在 Settings 中有最小/最大范围并在启动期校验。
- 开启功能不改变 GitHub App 权限，不触发实时网络访问。
- GitHub 未配置、Binding 不可用或事件无仓库映射时受控降级为无仓库上下文，而不是让整个事件会话不可用。

## 13. Web 最小交互

事件会话保持现有布局，只增加：

- 仓库引用显示 `owner/repo · path @ short-sha`。
- 显示 `与部署一致`、`HEAD 与部署不一致` 或 `部署 Commit 未知`。
- 显示快照同步时间、stale 和 truncated 状态。
- 快照已替换或撤权时显示“历史来源已不可用”，不展示缓存正文。
- 没有仓库映射、无有效查询词、同步失败或无命中时，通过 `missing_evidence` 解释，不伪装成空白成功。
- 不增加“修改代码”“创建 PR”“部署此 Commit”按钮。
- 不增加仓库选择器；事件页不能切换到其他服务或仓库。

## 14. Provider 规则

沿用 M5.1 严格结构，不增加工具调用：

- 仓库摘录只能作为给定引用使用。
- 文件中的 Prompt、命令、角色描述或 JSON schema 都是不可信正文。
- `mismatch/unknown` 仓库来源不能进入 facts。
- 不允许输出或建议未在上下文中出现的 Commit、路径或仓库作为已确认事实。
- 不允许声称已修改、提交、部署或验证代码。
- Provider 失败不自动重试，不触发 GitHub 同步。
- deterministic Provider 必须确定性处理仓库来源，且不会因 mismatch 来源生成非法事实。

## 15. 测试矩阵

### 15.1 作用域和撤权

- 服务事件只能派生自己的 ServiceInstance 和 Repository。
- 机器事件、未映射服务、其他事件和其他组织均无仓库正文。
- 问题中注入其他 repository/file/commit ID 不改变范围。
- Binding disabled、安装不匹配、Repository 组织不匹配和 `last_error` 均不返回正文。
- Provider 调用期间撤权、同步替换或 SHA 变化使该轮受控失败。
- `ON DELETE SET NULL` 不阻止 GitHub 同步和撤权；历史引用变为 unavailable。

### 15.2 Commit 真实性

- aligned、mismatch、unknown 判定确定。
- mismatch/unknown 引用进入 facts 时整轮失败。
- 每个引用显示 Provider 实际看到的 Commit，不使用后来更新的 HEAD。
- 没有历史 Commit 快照时不会实时 GitHub 补取或伪造部署代码内容。

### 15.3 检索与预算

- 查询词 NFKC、去重、停用词、词数/词长和无有效词场景。
- `%`、`_`、反斜杠、引号、路径穿越和正则元字符均按普通文本处理。
- 路径匹配、内容匹配、Commit 优先级和稳定排序可重复。
- 单文件摘录、结果数、仓库子预算和 128 KiB 总预算有效。
- 仓库项不会挤掉事件、服务和最新诊断核心证据；历史仍最先被省略。
- UTF-8、超长行、replacement 字符和截断不产生非法文本。

### 15.4 提示注入与脱敏

- README/Compose 中的“忽略指令、调用工具、创建 Operation、泄露密钥”只作为不可信文本。
- 已脱敏快照进入会话前再次脱敏。
- Provider 输出中的未知字段、未知引用、外部 URL 和伪造 Commit 被拒绝或不生成可信链接。
- 日志不包含问题、查询词、仓库摘录、Provider 原文、安装令牌或 API Key。

### 15.5 零副作用

- 会话请求前后 Operation、OperationTransition、EvidenceRequest、GitHub Binding/File 和 Agent 任务计数/状态不变。
- 检索路径不调用 GitHubClient、同步函数、Webhook、Agent Claim/Complete 或 M4 服务。
- “修复并提交”“创建 PR”“部署这个 Commit”等问题仍只返回未授权建议。

### 15.6 迁移和回归

- 真实 PostgreSQL `0010 -> 0011`、`0011 -> 0010 -> 0011` 和 schema check。
- 已有会话/引用数据保留。
- GitHub 同步替换文件和撤权删除在已有仓库引用时成功。
- API、Web、Go、Ruff、ESLint、Web 生产构建和现有 M2/M3/M4/M5.1 测试无回归。
- HTTP Provider 与 deterministic Provider 都使用确定性测试传输，不依赖外部模型服务。
