# M4.2 安全部署协议与验收

M4.2 把 M4 的受控写操作从"重启"扩展到"部署"：把一个已映射、Agent 本地明确授权、控制平面标记为非关键的单副本 Docker Compose 服务，从当前不可变镜像 digest 部署到**同一镜像仓库**的另一个不可变 digest，失败后只能通过另一笔明确确认的操作回滚到冻结的旧 digest。

M4.2 复用 M4 的 Operation 框架（状态机、Ed25519 签名、`active_key` 并发锁、审计、本地账本幂等），见 [M4_OPERATIONS.md](./M4_OPERATIONS.md) §2–4。本文件只写部署特有的扩展、威胁模型和分阶段验收。协议不向后兼容 M4 v1 任务：M4.2 使用独立的 v2 任务，旧 v0.4.0 Agent 只声明 restart 能力，收不到部署任务。

## 1. 范围与边界

唯一支持的范围：

> 一个明确授权的、非关键、单副本 Docker Compose 服务，从当前镜像 digest 部署到同一镜像仓库的另一个不可变 digest；失败后只能通过另一笔明确确认的操作回滚到冻结的旧 digest。

明确不做：

- Git pull、源码构建、仓库写入
- 可变 tag（例如单独使用 `latest`）；目标必须是完整 `repo@sha256:...`
- 修改 Compose 文件、环境变量、卷、网络或端口
- 多服务部署和依赖服务重建（强制 `--no-deps`）
- 任意路径、argv、命令或 Shell
- 未经确认的自动回滚

测试载体：保留 aliyun-VPS 的 `m4-canary` 继续承担重启回归；另建一个支持 A/B 镜像版本的 `m4-deploy-canary`（同仓库两个功能性镜像 digest）专做部署金丝雀。

## 2. 威胁模型

- **镜像供应链**：目标必须是完整 `repo@sha256:...`；同仓库校验（目标 registry 必须等于当前镜像 registry）；管理员显式授权具体 digest 并由控制平面 Ed25519 签名绑定；digest 内容寻址，pull 后不可被注册中心偷换。
- **Agent 执行**：固定 `docker` / `docker compose` 参数，不经过 Shell；服务名来自 stable service key 校验；digest 格式校验；`--no-deps` 只重建指定服务；临时 override 文件 `0600`、用后删除。
- **漂移检测**：确认时重新预检；执行前 Agent 再次核对当前 digest，与计划冻结的 current digest 不一致立即拒绝。
- **验证**：新鲜报告必须证明**实际运行 image digest == 目标 digest**（不只看健康），且 `healthy=true` 并持续稳定窗口。这能抓"重建了但跑的还是旧镜像"（pull 未生效、override 未应用、compose 用了旧 tag 等）。
- **回滚**：不自动回滚；回滚是独立确认操作，目标只能是原计划冻结的旧 digest。
- **凭据**：部署依赖宿主机**已有的 registry 登录**（`/root/.docker/config.json` 等），Agent 配置不保存 registry 凭据；同仓库约束保证现有凭据对目标 digest 适用。
- **Compose 完整性**：不修改原 Compose 文件；override 是 Agent 本地 ephemeral 文件。

## 3. 数据模型扩展（迁移 0007）

`0007_m4_2_deploy` 是 M4.1 之后第一次走标准 `migrate` 流程的加列迁移，顺便验证 M4.1 发布基础设施。新增字段：

- `operations.current_digest`（String，部署/回滚计划冻结的当前 digest）
- `operations.target_digest`（String，目标 digest）
- `operations.rollback_of`（String，可空，回滚操作指向的原部署操作 id）

`AgentOperationCapability` 复用现有结构，新增 `action_type = docker_compose_deploy`。`active_key`、`idempotency_key`、`task_nonce`、`signing_key_id`、状态机、审计转换等全部复用 M4。

## 4. 协议 v2 任务（不扩展 M4 v1）

M4 的 v1 任务用 `DisallowUnknownFields` 严格解码，且签名输入是固定 11 字段。M4.2 不能偷偷扩展 v1（会破坏旧 Agent 的严格解码和安全模型）。新增 v2 任务：

```
v2
operation_id
action_type            = docker_compose_deploy
agent_id
service_kind           = docker
service_key
current_digest         = repo@sha256:...
target_digest          = repo@sha256:...
issued_at
expires_at
idempotency_key
attempt
nonce
key_id
```

签名输入为上述固定顺序、换行连接。Agent 对 v2 任务用独立严格解码；`action_type` 和 `version` 不匹配 v1 的任务一律拒绝。旧 v0.4.0 Agent 不声明 `docker_compose_deploy` 能力，控制平面不会为其创建部署任务。

## 5. Agent 本地发现（M4.2a）

Agent 识别 Compose 部署候选并上报（**只上报摘要，不上报本地目标**）：

- stable service key
- 当前镜像仓库（registry host）
- 当前不可变 image digest：**RepoDigest**，取 `docker inspect <container>` 的 `.RepoDigests`（形如 `repo@sha256:...`），**不是**本地 image ID（`{{.Image}}` 的 sha256）
- 是否单副本
- 是否满足部署条件
- 不可部署时返回固定原因码（多副本、无 healthcheck、非 Compose 服务、digest 无法解析等）

以下内容**留在 Agent 本地**，不进入协议或数据库：

- Compose 文件路径、working directory
- 容器 ID、Docker target
- Registry 凭据

新增独立部署策略 `AGENT_DEPLOY_POLICY`（默认 `disabled`），与 `AGENT_OPERATION_POLICY` 分离。不能因为原来允许 `docker_restart` 就自动获得部署权限。安装器新增 `--deploy-policy docker-compose-deploy` 及对应 key id / 公钥（可复用 M4 的 Ed25519 key，或独立 key）。

## 6. 控制平面计划（M4.2a）

控制平面能创建并展示**冻结计划**，但不能确认、排队或执行：

- 当前 digest
- 目标 digest（M4.2a 阶段由**管理员输入**`repo@sha256:...`；后续 M4.2b+ 可接 GitHub App 读 release/workflow 自动取，留作扩展）
- 同仓库校验结果
- 影响的唯一 stable service key
- 健康验证条件
- 可回滚的旧 digest（= 当前 digest）
- 拒绝原因

**M4.2a 完成标准：即使 API 被调用，也不存在任何 pull、rebuild 或执行路径。** 部署任务 v2 即使被构造，Agent 端无 `docker_compose_deploy` 能力也不会领取。

## 7. 部署机制（M4.2b）

目标必须是完整 `repo@sha256:...`。流程：

1. 确认时重新预检（capability、`restart_enabled`/`deploy_enabled`、non_critical、observation_fresh、digest 可解析、同仓库、单副本、有 healthcheck）。
2. 执行前 Agent **再次核对当前 digest** == 计划冻结的 current_digest，漂移立即拒绝（`drift_rejected`）。
3. 签名任务绑定 current_digest + target_digest + agent + stable service key + 有效期 + nonce + 幂等键。
4. Agent 本地生成临时 override 文件（`0600`，用后删除）：
   ```yaml
   services:
     <service>:
       image: repo@sha256:<target_digest>
   ```
5. 固定命令链（无 Shell，固定参数）：
   ```
   docker pull repo@sha256:<target_digest>
   docker compose -f <原 compose 文件> -f <override> --project-directory <working_dir> up -d --no-deps --force-recreate <service>
   ```
   - compose 文件路径、working_dir 从容器 label（`com.docker.compose.project.config_files`、`com.docker.compose.project.working_dir`）本地解析。
   - `--no-deps` 只重建指定服务，不级联依赖。
   - `--force-recreate` 强制用目标镜像重建容器。
6. 独立执行超时 `operation_deploy_execution_timeout_seconds`（默认 **300s**，与 restart 的 30s 分开；pull + recreate 比 restart 慢得多）。
7. 输出继续限行、限字节、脱敏，且**不含本地路径、容器 target 或 compose 文件路径**。
8. 命令成功（exit 0）只进入 Verifying。

## 8. 健康验证（M4.2b）

新鲜 Agent 报告必须**同时**证明：

- 服务为 `running`
- `healthy` 为 `true`（目标镜像**必须有 healthcheck**；无 healthcheck 的镜像 `healthy=null`，不可部署，在发现阶段即返回原因码）
- 实际 image digest（RepoDigest）**== 目标 digest**
- 持续通过稳定窗口（复用 M4 的 `operation_verification_window_seconds` / `timeout_seconds`）

digest 不匹配或 unhealthy 超过验证截止时间 -> `failed`。结果未知时沿用 M4 保守策略：不自动重放（`execution_outcome_unknown`）。

## 9. 失败语义

显式区分三类失败：

- **pull 前失败**（digest 不在注册中心、网络、鉴权）：服务**未变更**（仍跑旧镜像），干净 `failed`，**不需要回滚**。
- **recreate 后不健康或 digest 不匹配**：服务**已变到 target**，需要回滚到旧 digest。
- **执行中崩溃**：`execution_outcome_unknown`，不自动重放（沿用 M4 本地账本）。

## 10. 显式回滚（M4.2c）

第一版**不自动回滚**。失败后生成一笔独立回滚计划：

- 目标只能是原计划冻结保存的旧 digest（`current_digest`）。
- 重新预检。
- 再次人工确认。
- 独立签名、执行、健康验证和审计（复用 §7 的 override 机制，target = 旧 digest）。
- 回滚执行前同样校验当前 digest：必须等于失败部署的 target_digest（即当前正在跑的坏镜像），漂移即拒。

这样避免"部署失败后系统未经确认又执行第二次写操作"。

## 11. 分阶段验收

**M4.2a（只读发现与计划）**：

- Agent 上报部署候选（stable key、当前 RepoDigest、单副本、条件、原因码），不上传本地路径/target/凭据。
- 控制平面创建并展示冻结计划，但不能确认/排队/执行。
- 协议 v2 兼容：旧 v0.4.0 Agent 不声明部署能力、收不到部署任务；v2 任务严格解码。
- **完成标准：不存在任何 pull/rebuild/execute 路径。**

**M4.2b（受控执行）**：

- 目标 `repo@sha256:...`，确认重预检，执行前 Agent 校验当前 digest 漂移即拒。
- 签名绑定 current+target digest；override 文件机制；`--no-deps --force-recreate`；300s 超时。
- 验证：running + healthy + 实际 digest == 目标 digest + 稳定窗口。
- 隔离验证：A->B 成功；B unhealthy -> `failed`；digest 不匹配 -> `failed`；同服务并发 409；漂移拒绝；重复 Complete 幂等。

**M4.2c（显式回滚）**：

- 失败后独立回滚计划（目标=冻结旧 digest），重预检 + 再确认 + 独立签名/执行/验证/审计。
- 隔离验证：A->B(失败)->A 回滚成功；回滚漂移拒绝。

**生产金丝雀**：aliyun-VPS `m4-deploy-canary` 做 A -> B -> A 全闭环（含 B 为坏镜像的失败回滚路径）。
