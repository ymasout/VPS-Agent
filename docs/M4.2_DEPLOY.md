# M4.2 安全部署协议与验收

M4.2 把 M4 的受控写操作从"重启"扩展到"部署"：把一个已映射、Agent 本地明确授权、控制平面标记为非关键的单副本 Docker Compose 服务，从当前镜像 digest 部署到**同一完整仓库**（`registry/repo`，例如 `ghcr.io/org/app`）的另一个不可变 digest，失败后只能通过另一笔明确确认的操作回滚到冻结的旧 digest。

M4.2 复用 M4 的 Operation 框架（状态机、Ed25519 签名、`active_key` 并发锁、审计、本地账本幂等），见 [M4_OPERATIONS.md](./M4_OPERATIONS.md) §2–4。**协议 v2 任务、签名、pull、Compose 调用和回滚全部推迟到 M4.2b/M4.2c**；M4.2a 是纯只读发现与计划展示，不存在任何执行路径。旧 v0.4.0 Agent 不上报部署候选、收不到部署任务，不受影响。

## 1. 范围与边界

唯一支持的范围：

> 一个明确授权的、非关键、单副本 Docker Compose 服务，从当前镜像 digest 部署到**同一完整仓库**（`registry/repo` 完全相同）的另一个不可变 digest；失败后只能通过另一笔明确确认的操作回滚到冻结的旧 digest。

明确不做：

- Git pull、源码构建、仓库写入
- 可变 tag（例如单独使用 `latest`）；目标必须是完整 `repo@sha256:<64 位十六进制>`
- 跨仓库部署（即使同一 registry）；目标仓库必须与当前镜像的标准化完整仓库**完全相同**
- 修改 Compose 文件、环境变量、卷、网络或端口
- 多服务部署和依赖服务重建（强制 `--no-deps`）
- 任意路径、argv、命令或 Shell
- 未经确认的自动回滚

测试载体：保留 aliyun-VPS 的 `m4-canary` 继续承担重启回归；另建一个支持 A/B 镜像版本的 `m4-deploy-canary`（同完整仓库两个功能性镜像 digest）专做部署金丝雀。

## 2. 威胁模型

- **镜像供应链**：目标必须是完整 `repo@sha256:<64 位十六进制>`（严格格式校验）；**同完整仓库校验**——目标的标准化 `registry/repo` 必须与当前镜像的标准化 `registry/repo` 完全相同，不只比 registry host（否则会允许从同一 registry 的其他仓库部署）；管理员显式授权具体 digest 并由控制平面 Ed25519 签名绑定；digest 内容寻址，pull 后不可被注册中心偷换。
- **Agent 执行**：固定 `docker` / `docker compose` 参数，不经过 Shell；服务名来自 stable service key 校验；digest 格式校验；`--no-deps` 只重建指定服务；临时 override 文件 `0600`、用后删除。
- **漂移检测**：确认时重新预检；执行前 Agent 再次核对当前 digest，与计划冻结的 current digest 不一致立即拒绝。
- **验证**：新鲜报告必须证明**实际运行 image digest == 目标 digest**（不只看健康），且 `healthy=true` 并持续稳定窗口。这能抓"重建了但跑的还是旧镜像"。
- **回滚**：不自动回滚；回滚是独立确认操作，目标只能是原计划冻结的旧 digest。
- **凭据**：部署依赖宿主机**已有的 registry 登录**，Agent 配置不保存 registry 凭据；同完整仓库约束保证现有凭据对目标 digest 适用。
- **Compose 完整性**：不修改原 Compose 文件；override 是 Agent 本地 ephemeral 文件；Compose label 中的路径不能直接信任，必须限制在 Agent 本地配置的允许目录内，做规范化路径、软链接越界和多文件顺序检查（M4.2b 实现时落地）。

## 3. 分阶段总览与数据模型

M4.2 分三阶段，每阶段只引入该阶段实际使用的结构，避免迁移混入尚未启用的写能力：

- **M4.2a（只读发现与计划）**：迁移 `0007_m4_2_deploy_candidates` 仅新增独立的 `agent_deployment_candidates` 表（见 §4.1）。**不**给 `operations` 加任何 digest 字段，**不**加 `deploy_enabled`，**不**实现协议 v2。
- **M4.2b（受控执行）**：迁移 `0008_m4_2_deploy_execute` 给 `operations` 加 `current_digest`、`target_digest`，给 `service_instances` 加 `deploy_enabled`；实现协议 v2 任务、签名、pull、Compose 重建和健康验证。
- **M4.2c（显式回滚）**：迁移 `0009_m4_2_rollback` 给 `operations` 加 `rollback_of`；实现独立回滚计划。

M4.2b/M4.2c 复用 M4 的 `active_key`、`idempotency_key`、`task_nonce`、`signing_key_id`、状态机和审计转换。

## 4. M4.2a：只读发现与计划（无执行路径）

### 4.1 Agent 候选发现与上报

Agent 识别 Compose 部署候选并上报（**只上报摘要，不上报本地目标**）。**使用独立的 `AgentDeploymentCandidate` 模型和上报字段，不复用 `AgentOperationCapability`**——后者表示"Agent 已授权的写操作能力"，且每次上报整体替换，既不能保存 repository/digest/资格/原因码，也不应在 Agent 尚无部署执行器时就宣告写能力。

`agent_deployment_candidates` 表（每次上报按 agent 整体替换，类似 `agent_evidence_sources`）：

- `agent_id`、`service_kind`、`service_key`（stable key）
- `repository`（标准化完整仓库 `registry/repo`）
- `current_digest`（`repo@sha256:<64hex>`）
- `eligible`（bool）
- `reason_code`（不可部署时的固定原因码：多副本、无 healthcheck、非 Compose 服务、digest 无法解析、digest 歧义、仓库无法标准化等）
- `observed_at`

**当前 digest 采集（两步，不能直接用容器 inspect 的 `.RepoDigests`，容器 inspect 没有该字段）**：

1. `docker inspect <container>` 取 `.Image`（image ID）和 `.Config.Image`（原始镜像名，如 `ghcr.io/org/app:tag` 或 `ghcr.io/org/app@sha256:...`）。
2. `docker image inspect <image ID>` 取 `.RepoDigests`（数组，形如 `ghcr.io/org/app@sha256:...`）。
3. 从 `.RepoDigests` 中选择**唯一**匹配当前标准化完整仓库的项；**缺失或歧义（0 个或 >1 个匹配）均判为不可部署**（原因码 `digest_unresolved` / `digest_ambiguous`）。

以下内容**留在 Agent 本地**，不进入协议或数据库：Compose 文件路径、working directory、容器 ID、Docker target、Registry 凭据。

新增独立部署策略 `AGENT_DEPLOY_POLICY`，与 `AGENT_OPERATION_POLICY` 分离，默认 `disabled`：

- **M4.2a 只允许 `plan_only`**：只做发现与候选上报，不构成任何写能力授权。
- **`docker_compose_deploy`（执行）属于 M4.2b**，且必须由 M4.2b 显式设置；从 M4.2a 升级到 M4.2b 时，`plan_only` 保持 `plan_only`，**不会静默获得写权限**，必须人工改成 `docker_compose_deploy` 才启用执行。

不能因为原来允许 `docker_restart` 就自动获得部署权限。

### 4.2 控制平面计划（只展示，不可确认）

控制平面能创建并展示**冻结计划**，但不能确认、排队或执行：

- 当前 digest（来自候选）
- 目标 digest（M4.2a 阶段由**管理员输入**`repo@sha256:<64hex>`；后续可接 GitHub App 读 release/workflow 自动取，留作扩展）
- **同完整仓库校验**：目标的标准化 `registry/repo` 必须与候选的 `repository` 完全相同
- 影响的唯一 stable service key
- 健康验证条件
- 可回滚的旧 digest（= 当前 digest，M4.2b 起用）
- 拒绝原因

**显式拒绝 deploy action**：现有 M4 的确认、排队、领取、执行接口（`POST /operations`、`/operations/{id}/confirm`、`/agents/operations/next`、`/start`、`/complete`）必须**显式拒绝** `docker_compose_deploy` action，不能依赖"当前代码恰好只处理 restart"。即使 schema 层已限制，仍加运行时校验作为纵深防御，确保 M4.2b 启用执行前任何 deploy action 都无法走执行路径。

**M4.2a 完成标准**：不存在任何协议 v2 任务、签名、领取、pull、Compose 调用或回滚代码路径。即使 API 被调用，也不产生写操作。

## 5. M4.2b：受控执行（协议 v2 + 部署机制）

M4.2a 通过并审计后实现。协议 v2、签名任务、pull、Compose 重建、健康验证在此阶段引入。

### 5.1 协议 v2 任务（不扩展 M4 v1）

M4 的 v1 任务用 `DisallowUnknownFields` 严格解码，签名输入是固定字段。M4.2b 新增 v2 任务（独立严格解码），不能偷偷扩展 v1：

```
v2
operation_id
action_type            = docker_compose_deploy
agent_id
service_kind           = docker
service_key
current_digest         = repo@sha256:<64hex>
target_digest          = repo@sha256:<64hex>
issued_at
expires_at
idempotency_key
attempt
nonce
key_id
```

签名输入为上述固定顺序、换行连接。`action_type`/`version` 不匹配 v1 的任务一律拒绝。控制平面只为声明了 `docker_compose_deploy` 能力的 Agent 创建部署任务（该能力由 M4.2b 的 `AGENT_DEPLOY_POLICY=docker_compose_deploy` 显式启用）。

### 5.2 部署机制

目标必须是完整 `repo@sha256:<64hex>`。流程：

1. 确认时重新预检（capability、`deploy_enabled`、non_critical、observation_fresh、digest 可解析、**同完整仓库**、单副本、有 healthcheck）。
2. 执行前 Agent **再次核对当前 digest** == 计划冻结的 current_digest，漂移立即拒绝（`drift_rejected`）。
3. 签名任务绑定 current_digest + target_digest + agent + stable service key + 有效期 + nonce + 幂等键。
4. Agent 本地生成临时 override 文件（`0600`，用后删除），路径限制在 Agent 本地配置的允许目录内，做规范化与越界检查：
   ```yaml
   services:
     <service>:
       image: repo@sha256:<target_digest>
   ```
5. 固定命令链（无 Shell，固定参数）：
   ```
   docker pull repo@sha256:<target_digest>
   docker compose -f <原 compose 文件> -f <override> --project-directory <working_dir> up -d --no-deps --force-recreate <service>
   ```
   - compose 文件路径、working_dir 从容器 label（`com.docker.compose.project.config_files`、`com.docker.compose.project.working_dir`）本地解析，且必须落在允许目录内。
   - `--no-deps` 只重建指定服务，不级联依赖。
   - `--force-recreate` 强制用目标镜像重建容器。
6. 独立执行超时 `operation_deploy_execution_timeout_seconds`（默认 **300s**，与 restart 的 30s 分开；pull + recreate 比 restart 慢得多）。
7. 输出继续限行、限字节、脱敏，且**不含本地路径、容器 target 或 compose 文件路径**。
8. 命令成功（exit 0）只进入 Verifying。

### 5.3 健康验证

新鲜 Agent 报告必须**同时**证明：

- 服务为 `running`
- `healthy` 为 `true`（目标镜像**必须有 healthcheck**；无 healthcheck 的镜像 `healthy=null`，不可部署，在发现阶段即返回原因码）
- 实际 image digest（§4.1 的两步采集得到的 RepoDigest）**== 目标 digest**
- 持续通过稳定窗口（复用 M4 的 `operation_verification_window_seconds` / `timeout_seconds`）

digest 不匹配或 unhealthy 超过验证截止时间 -> `failed`。结果未知时沿用 M4 保守策略：不自动重放（`execution_outcome_unknown`）。

### 5.4 失败语义

显式区分三类失败：

- **pull 前失败**（digest 不在注册中心、网络、鉴权）：服务**未变更**（仍跑旧镜像），干净 `failed`，**不需要回滚**。
- **recreate 后不健康或 digest 不匹配**：服务**已变到 target**，需要回滚到旧 digest。
- **执行中崩溃**：`execution_outcome_unknown`，不自动重放（沿用 M4 本地账本）。

## 6. M4.2c：显式回滚

第一版**不自动回滚**。失败后生成一笔独立回滚计划：

- 目标只能是原计划冻结保存的旧 digest（`current_digest`）。
- 重新预检。
- 再次人工确认。
- 独立签名、执行、健康验证和审计（复用 §5.2 的 override 机制，target = 旧 digest）。
- 回滚执行前同样校验当前 digest：必须等于失败部署的 target_digest（即当前正在跑的坏镜像），漂移即拒。

这样避免"部署失败后系统未经确认又执行第二次写操作"。

## 7. 分阶段验收

**M4.2a（只读发现与计划）**：

- `AgentDeploymentCandidate` 独立表 + 上报字段；Agent 上报 stable key、完整仓库、当前 RepoDigest（两步采集）、单副本、资格、原因码；不上传本地路径/target/凭据。
- `AGENT_DEPLOY_POLICY` 仅支持 `plan_only`（默认 `disabled`）；`docker_compose_deploy` 不存在。
- 控制平面创建并展示冻结计划，目标必须与候选的标准化完整仓库完全相同；计划不可确认/排队/执行。
- 现有确认/排队/领取/执行接口显式拒绝 deploy action。
- **不实现**协议 v2、签名任务、pull、Compose 调用、回滚。
- **完成标准：不存在任何执行路径。** 旧 v0.4.0 Agent 不上报候选、不受影响。

**M4.2b（受控执行）**：

- 协议 v2 任务（独立严格解码，旧 Agent 不收）；`AGENT_DEPLOY_POLICY=docker_compose_deploy` 显式启用。
- 目标 `repo@sha256:<64hex>`，确认重预检，执行前 Agent 校验当前 digest 漂移即拒。
- 签名绑定 current+target digest；override 文件机制；`--no-deps --force-recreate`；300s 超时。
- 验证：running + healthy + 实际 digest == 目标 digest + 稳定窗口。
- 隔离验证：A->B 成功；B unhealthy -> `failed`；digest 不匹配 -> `failed`；同服务并发 409；漂移拒绝；重复 Complete 幂等。

**M4.2c（显式回滚）**：

- 失败后独立回滚计划（目标=冻结旧 digest），重预检 + 再确认 + 独立签名/执行/验证/审计。
- 隔离验证：A->B(失败)->A 回滚成功；回滚漂移拒绝。

**生产金丝雀**：aliyun-VPS `m4-deploy-canary` 做 A -> B -> A 全闭环（含 B 为坏镜像的失败回滚路径）。
