# M4.2 安全部署协议与验收

M4.2 把 M4 的受控写操作从"重启"扩展到"部署"：把一个已映射、Agent 本地明确授权、控制平面标记为非关键的单副本 Docker Compose 服务，从当前镜像 digest 部署到**同一完整仓库**（`registry/repo`，例如 `ghcr.io/org/app`）的另一个不可变 digest，失败后只能通过另一笔明确确认的操作回滚到冻结的旧 digest。

M4.2 复用 M4 的 Operation 框架（状态机、Ed25519 签名、`active_key` 并发锁、审计、本地账本幂等），见 [M4_OPERATIONS.md](./M4_OPERATIONS.md) §2–4。M4.2a 是纯只读发现与计划展示，不存在任何执行路径；M4.2b 新增协议 v2、签名、pull 和单服务 Compose 重建，M4.2c 再增加显式回滚。旧 v0.4.0 Agent 不上报部署候选、收不到部署任务；已运行 `v0.4.1 plan_only` 的 Agent 升级后也不会静默获得写权限。

## 1. 范围与边界

唯一支持的范围：

> 一个明确授权的、非关键、单副本 Docker Compose 服务，从当前镜像 digest 部署到**同一完整仓库**（`registry/repo` 完全相同）的另一个不可变 digest；失败后只能通过另一笔明确确认的操作回滚到冻结的旧 digest。

明确不做：

- Git pull、源码构建、仓库写入
- 可变 tag（例如单独使用 `latest`）；目标必须是完整 `repo@sha256:<64 位十六进制>`
- 跨仓库部署（即使同一 registry）；目标仓库必须与当前镜像的标准化完整仓库**完全相同**
- 修改 Compose 文件、环境变量、卷、网络或端口
- 多服务部署和依赖服务重建（强制 `--no-deps`）
- 任意路径、argv、命令或 Shell
- 未经确认的自动回滚

测试载体：保留 aliyun-VPS 的 `m4-canary` 继续承担重启回归；另建一个支持 A/B 镜像版本的 `m4-deploy-canary`（同完整仓库两个功能性镜像 digest）专做部署金丝雀。

## 2. 威胁模型

- **镜像供应链**：目标必须是完整 `repo@sha256:<64 位十六进制>`（严格格式校验）；**同完整仓库校验**——目标的标准化 `registry/repo` 必须与当前镜像的标准化 `registry/repo` 完全相同，不只比 registry host（否则会允许从同一 registry 的其他仓库部署）；管理员显式授权具体 digest 并由控制平面 Ed25519 签名绑定；digest 内容寻址，pull 后不可被注册中心偷换。
- **Agent 执行**：固定 `docker` / `docker compose` 参数，不经过 Shell；服务名来自 stable service key 校验；digest 格式校验；`--no-deps` 只重建指定服务；临时 override 文件 `0600`、用后删除。
- **漂移检测**：确认时重新预检；执行前 Agent 再次核对当前 digest，与计划冻结的 current digest 不一致立即拒绝。
- **验证**：新鲜报告必须证明**实际运行 image digest == 目标 digest**（不只看健康），且 `healthy=true` 并持续稳定窗口。这能抓"重建了但跑的还是旧镜像"。
- **回滚**：不自动回滚；回滚是独立确认操作，目标只能是原计划冻结的旧 digest。
- **凭据**：部署依赖宿主机**已有的 registry 登录**，Agent 配置不保存 registry 凭据；同完整仓库约束保证现有凭据对目标 digest 适用。
- **Compose 完整性**：不修改原 Compose 文件；override 是 Agent 本地 ephemeral 文件；Compose label 中的路径不能直接信任，必须限制在 Agent 本地配置的允许目录内，做规范化路径、软链接越界和多文件顺序检查（M4.2b 实现时落地）。

## 3. 分阶段总览与数据模型

M4.2 分三阶段，每阶段只引入该阶段实际使用的结构，避免迁移混入尚未启用的写能力：

- **M4.2a（只读发现与计划）**：迁移 `0007_m4_2_deploy_candidates` 仅新增独立的 `agent_deployment_candidates` 表（见 §4.1）。冻结计划复用现有 `operations.plan_snapshot` JSON，不增加 digest 列；快照固定写入 `plan_version=m4.2a-plan-only-v1`、`execution_policy=none`、`permanently_non_executable=true`，且 `active_key=null`。**不**加 `deploy_enabled`，**不**实现协议 v2。
- **M4.2b（受控执行）**：迁移 `0008_m4_2_deploy_execute` 给 `operations` 加 `current_digest`、`target_digest`（均可空，兼容 restart 操作和历史 M4.2a 计划），给 `service_instances` 加 `deploy_enabled`（`NOT NULL DEFAULT false`，须显式启用，与 `restart_enabled` 独立）。**不从 M4.2a 的 `plan_snapshot` JSON 回填任何可执行字段**；只有新创建的 `m4.2b-executable-v1` 计划能获得 `active_key`、确认和签名，M4.2a 的 `m4.2a-plan-only-v1` 计划保持永久不可执行。实现协议 v2 任务、签名、pull、Compose 重建和健康验证。
- **M4.2c（显式回滚）**：迁移 `0009_m4_2_rollback` 给 `operations` 加 `rollback_of`；实现独立回滚计划。

M4.2b/M4.2c 复用 M4 的 `active_key`、`idempotency_key`、`task_nonce`、`signing_key_id`、状态机和审计转换。

## 4. M4.2a：只读发现与计划（无执行路径）

### 4.1 Agent 候选发现与上报

Agent 识别 Compose 部署候选并上报（**只上报摘要，不上报本地目标**）。**使用独立的 `AgentDeploymentCandidate` 模型和上报字段，不复用 `AgentOperationCapability`**——后者表示"Agent 已授权的写操作能力"，且每次上报整体替换，既不能保存 repository/digest/资格/原因码，也不应在 Agent 尚无部署执行器时就宣告写能力。

`agent_deployment_candidates` 表（每次上报按 agent 整体替换，类似 `agent_evidence_sources`）：

- `agent_id`、`service_kind`、`service_key`（stable key）
- `repository`（标准化完整仓库 `registry/repo`；无法解析时为空）
- `current_digest`（`repo@sha256:<64hex>`；无法解析或歧义时为空）
- `eligible`（bool）
- `reason_code`（不可部署时的固定原因码：多副本、无 healthcheck、非 Compose 服务、digest 无法解析、digest 歧义、仓库无法标准化等）
- `observed_at`（控制平面接受该报告的服务器时间，不信任 Agent 自报的未来时间来延长新鲜度）

**当前 digest 采集（两步，不能直接用容器 inspect 的 `.RepoDigests`，容器 inspect 没有该字段）**：

1. `docker inspect <container>` 取 `.Image`（image ID）和 `.Config.Image`（原始镜像名，如 `ghcr.io/org/app:tag` 或 `ghcr.io/org/app@sha256:...`）。
2. `docker image inspect <image ID>` 取 `.RepoDigests`（数组，形如 `ghcr.io/org/app@sha256:...`）。
3. 从 `.RepoDigests` 中选择**唯一**匹配当前标准化完整仓库的项（完全相同的重复值先去重）；**缺失或歧义（0 个或 >1 个不同匹配）均判为不可部署**（原因码 `digest_unresolved` / `digest_ambiguous`）。

仓库标准化在 Agent 与 API 使用同一组规则和测试向量：去掉 tag/digest 后识别 registry；缺省 registry 归一为 `docker.io`，`index.docker.io` 归一为 `docker.io`，Docker Hub 单段仓库补 `library/`；registry 和仓库路径转小写；拒绝 URL scheme、空路径段、空白和非法字符。示例：`ubuntu:24.04 -> docker.io/library/ubuntu`、`index.docker.io/library/nginx -> docker.io/library/nginx`、`ghcr.io/Org/App:v1 -> ghcr.io/org/app`、`localhost:5000/team/api:tag -> localhost:5000/team/api`。管理员目标必须已经是规范形式，不做静默纠正，并严格匹配 `repo@sha256:<64 个小写 hex>`。

固定原因码及判定优先级为：`inspect_failed`、`repository_unresolved`、`digest_unresolved` / `digest_ambiguous`、`multiple_replicas`、`healthcheck_missing`；M4.2b 本地执行预检还可收敛为 `compose_metadata_invalid`、`compose_path_untrusted`、`compose_config_drift` 或 `drift_rejected`。只上报 Compose 容器；非 Compose 服务不进入候选表。每次报告最多 128 个候选，控制平面按 Agent 整体替换；策略关闭或旧 Agent 缺少该可选字段时为空列表并清理旧候选。

以下内容**留在 Agent 本地**，不进入协议或数据库：Compose 文件路径、working directory、容器 ID、Docker target、Registry 凭据。

新增独立部署策略 `AGENT_DEPLOY_POLICY`，与 `AGENT_OPERATION_POLICY` 分离，默认 `disabled`：

- **M4.2a 只允许 `plan_only`**：只做发现与候选上报，不构成任何写能力授权。
- **`docker_compose_deploy`（执行）属于 M4.2b**，且必须由 M4.2b 显式设置；从 M4.2a 升级到 M4.2b 时，`plan_only` 保持 `plan_only`，**不会静默获得写权限**，必须人工改成 `docker_compose_deploy` 才启用执行。

不能因为原来允许 `docker_restart` 就自动获得部署权限。

### 4.2 控制平面计划（只展示，不可确认）

控制平面能创建并展示**冻结计划**，但不能确认、排队或执行：

- 当前 digest（来自候选）
- 目标 digest（M4.2a 阶段由**管理员输入**`repo@sha256:<64hex>`；后续可接 GitHub App 读 release/workflow 自动取，留作扩展）
- **同完整仓库校验**：目标的标准化 `registry/repo` 必须与候选的 `repository` 完全相同
- 影响的唯一 stable service key
- 健康验证条件
- 可回滚的旧 digest（= 当前 digest，M4.2b 起用）
- 拒绝原因

创建计划还要求同一份最新 Agent 报告中的映射服务当前为 `running` 且 `healthy=true`；候选镜像可解析但服务已停止、未知或不健康时，页面和 API 都拒绝创建计划。

冻结计划保存为 `action_type=docker_compose_deploy`、`status=planned` 的 Operation，只把 repository、current/target digest、stable service key、候选观测时间和未来验证条件写入 `plan_snapshot`；它不占用写操作 `active_key`，不生成 nonce、签名或任务。M4.2b **不得把 M4.2a 历史计划升级为可执行计划**，管理员必须基于当时的新鲜候选重新创建 M4.2b 计划。

**历史计划永久拒绝执行**：通用 `POST /operations` 继续只接受 restart；M4.2b 另设 `POST /deployment-operations`，只创建带 `m4.2b-executable-v1` 标记、独立 digest 列和 `active_key` 的新操作。确认、领取、开始、完成接口仅接受该标记或既有 restart；M4.2a 的 `m4.2a-plan-only-v1` 即使被异常改成 queued，也会在领取端终止且不下发任务。

**M4.2a 完成标准**：不存在任何协议 v2 任务、签名、领取、pull、Compose 调用或回滚代码路径。即使 API 被调用，也不产生写操作。

## 5. M4.2b：受控执行（协议 v2 + 部署机制）

M4.2a 已通过生产审计；本节的协议 v2、签名任务、pull、Compose 重建和健康验证已完成本地实现与隔离验证，尚未部署生产。

### 5.1 协议 v2 任务（不扩展 M4 v1）

M4 的 v1 任务用 `DisallowUnknownFields` 严格解码，签名输入是固定字段。M4.2b 新增 v2 任务（独立严格解码），不能偷偷扩展 v1：

```
v2
operation_id
action_type            = docker_compose_deploy
agent_id
service_kind           = docker
service_key
current_digest         = repo@sha256:<64hex>
target_digest          = repo@sha256:<64hex>
issued_at
expires_at
idempotency_key
attempt
nonce
key_id
```

签名输入为上述固定顺序、换行连接。`action_type`/`version` 不匹配 v1 的任务一律拒绝。控制平面只为声明了 `docker_compose_deploy` 能力的 Agent 创建部署任务（该能力由 M4.2b 的 `AGENT_DEPLOY_POLICY=docker_compose_deploy` 显式启用）。

### 5.2 部署机制

目标必须是完整 `repo@sha256:<64hex>`。流程：

1. 确认时重新预检（capability、`deploy_enabled`、non_critical、observation_fresh、digest 可解析、**同完整仓库**、单副本、有 healthcheck）。
2. 执行前 Agent **再次核对当前 digest** == 计划冻结的 current_digest，漂移立即拒绝（`drift_rejected`）。
3. 签名任务绑定 current_digest + target_digest + agent + stable service key + 有效期 + nonce + 幂等键。
4. Agent 本地生成临时 override 文件（`0600`，用后删除），路径限制在 Agent 本地配置的允许目录内，做规范化与越界检查：
   ```yaml
   services:
     <service>:
       image: repo@sha256:<target_digest>
   ```
5. **Compose 配置漂移门**：`up --force-recreate` 会重新应用当前 Compose 配置；若文件、`.env` 或变量已变化，可能同时改端口/卷/环境变量，违反"只换镜像"。执行前按 label 记录的原始 `-f` 顺序、project 和 working directory 计算两份候选 hash：`H_original` 使用原文件且不加 override；`H_current` 使用原文件加一份 Agent 临时生成的 `image=<current_digest>` 当前镜像 override。目标容器的 `com.docker.compose.config-hash` label 必须存在且等于 `H_original` 或 `H_current`，否则以 `compose_config_drift` 拒绝。前者接管尚未由 Agent 部署过的原生 Compose 容器，后者允许 Agent 部署后的下一次部署或回滚复现当前配置；真正的 `target_digest` override 永不参与漂移预检。Compose 会把本次临时 override 追加到新容器的 `project.config_files` label，因此后续预检只允许剥离**末尾唯一一项**严格匹配 Agent 状态目录下 `deploy-*/image-target.yaml` 的已删除临时文件；其他新增、缺失或越界路径仍失败关闭。两份 override 均为 `0600`、用后删除，任何文件顺序、环境解析或 hash 无法复现都失败关闭。
6. 固定命令链（无 Shell，固定参数）：
   ```
   docker pull repo@sha256:<target_digest>
   docker compose -f <原 compose 文件> -f <override> --project-name <label.project> --project-directory <working_dir> up -d --no-deps --no-build --pull never --force-recreate --scale <service>=1 <service>
   ```
   - compose 文件路径、working_dir、project 从容器 label（`com.docker.compose.project.config_files`、`com.docker.compose.project.working_dir`、`com.docker.compose.project`）本地解析，且路径必须落在 Agent 本地允许目录内。
   - `--project-name` 必须显式取自容器 label 的 project，确保重建同一项目下的目标容器，而非按文件名或 `COMPOSE_PROJECT_NAME` 默认出另一套项目。
   - `--no-build --pull never`：目标 digest 已由独立 `docker pull` 获取，Compose 阶段禁止隐式构建或二次拉取（避免原文件 `build:` 或 `pull_policy:` 触发额外动作）。
   - `--scale <service>=1`：Compose 的配置 hash 不绑定 scale/deploy replicas；执行命令必须显式维持单副本，防止原文件副本配置漂移令本次部署扩容。
   - `--no-deps` 只重建指定服务，不级联依赖。
   - `--force-recreate` 强制用 override 中的目标镜像重建容器。
7. 独立执行超时 `operation_deploy_execution_timeout_seconds`（默认 **300s**，与 restart 的 30s 分开；pull + recreate 比 restart 慢得多）。
8. 输出继续限行、限字节、脱敏，且**不含本地路径、容器 target 或 compose 文件路径**。
9. 命令成功（exit 0）只进入 Verifying。

### 5.3 健康验证

**同一份执行后的新鲜 Agent 报告**必须**同时**证明（不得把新健康状态与旧候选 digest 拼接，否则会把"旧镜像恢复健康"误判为部署成功）：

- 服务为 `running`
- `healthy` 为 `true`（目标镜像**必须有 healthcheck**；无 healthcheck 的镜像 `healthy=null`，不可部署，在发现阶段即返回原因码）
- **该报告中**的实际 image digest（§4.1 的两步采集得到的 RepoDigest）**== 目标 digest**（证明容器确实跑在新镜像上）
- 持续通过稳定窗口（复用 M4 的 `operation_verification_window_seconds` / `timeout_seconds`）

digest 不匹配或 unhealthy 超过验证截止时间 -> `failed`。结果未知时沿用 M4 保守策略：不自动重放（`execution_outcome_unknown`）。

### 5.4 失败语义

显式区分三类失败：

- **pull 前失败**（digest 不在注册中心、网络、鉴权）：服务**未变更**（仍跑旧镜像），干净 `failed`，**不需要回滚**。
- **recreate 后不健康或 digest 不匹配**：服务**已变到 target**，需要回滚到旧 digest。
- **执行中崩溃**：`execution_outcome_unknown`，不自动重放（沿用 M4 本地账本）。

## 6. M4.2c：显式回滚

第一版**不自动回滚，也不在失败时后台自动创建回滚计划**。管理员只能对一笔已进入终态 `failed`、已开始过执行且不是回滚操作的 `m4.2b-executable-v1` 部署，显式创建一笔独立回滚计划：

- 回滚计划使用新的 operation ID、幂等键、`active_key` 和 `rollback_of`；原失败部署保持不可变，且禁止对回滚再创建回滚链。
- 当前 digest 冻结为原失败部署的 `target_digest`，目标只能由控制平面取原计划保存的 `current_digest`；接口不接受管理员另行输入回滚目标。
- 创建与确认时都重新预检 Agent 在线、Compose 部署能力、控制面授权、non-critical、候选资格/新鲜度和同完整仓库。坏版本允许当前服务 unhealthy 或 stopped，回滚前不要求它已经健康；但服务映射与新鲜候选必须存在。
- 创建与确认时均要求候选实际 current digest == 原失败部署的 `target_digest`。若 pull 前即失败、服务仍在旧 digest，或其后已被人工改动，均以漂移拒绝，不创建/不签发回滚。
- 再次人工确认后才独立签名、执行、健康验证和审计；执行复用 §5.2 的协议 v2 与 override 机制（target = 原旧 digest），Agent 执行前仍会再次核对 current digest。
- 回滚成功的判断仍是同一份新鲜报告同时满足 target digest、`running`、`healthy=true` 和稳定窗口，不以 Compose 命令退出 0 代替健康验证。

这样避免"部署失败后系统未经确认又执行第二次写操作"。

## 7. 分阶段验收

**M4.2a（只读发现与计划）**：

- `AgentDeploymentCandidate` 独立表 + 上报字段；Agent 上报 stable key、完整仓库、当前 RepoDigest（两步采集）、单副本、资格、原因码；不上传本地路径/target/凭据。
- `AGENT_DEPLOY_POLICY` 仅支持 `plan_only`（默认 `disabled`）；`docker_compose_deploy` 不存在。
- 控制平面使用现有 `operations.plan_snapshot` 创建并展示永久不可执行的冻结计划，目标必须与候选的标准化完整仓库完全相同；计划不可确认/排队/执行，M4.2b 必须重新创建计划。
- 现有确认/排队/领取/执行接口显式拒绝 deploy action。

## 12. M4.2c 本地隔离验证（2026-07-22）
